# 007.26.07.01.AGX-2026-Priorities-Parse

## TL;DR

AGX (Agentic Fabric Platform) is the engineering substrate on which HashiCorp's managed Terraform agents will run. The memo outlines 2026 priorities for the platform team, owned by Marc Cosentino. AGX currently integrates with HCP-TF only. The 2026 work is about making the platform enterprise-ready: TFE support, multi-tenancy, richer trigger types, authenticated Terraform MCP and CLI connectors, and execution scalability.

The memo defines eleven customer use cases the platform is being built to serve - six core and five additional. These map directly to level-two workflows in Omar's ladder (004) and to the use cases named in the customer key asks document (005). The platform is not yet capable of supporting most of them: MCP currently uses static credentials, TFE integration does not yet exist, and multi-tenancy is still in progress.

The open decision on bring-your-own provider vs. HashiCorp-managed provider is unresolved and directly matches the architecture question Omar flagged in the 004 sync. This decision gates billing design, enterprise adoption, and customer privacy controls.

## Key Discussion Points

### 1. AGX vision

AGX is IPG's central platform for Managed Agents. Goal: enable teams to quickly build, govern, and embed agentic capabilities into real product workflows. The platform provides reusable infrastructure for agent orchestration, security, policy, and observability across both HCP Terraform and Terraform Enterprise.

**Definition of a Managed Agent (from the memo):**
> A managed agent is an AI agent designed, hosted, operated, and maintained by HashiCorp/IBM as part of the Terraform product experience. It runs within HashiCorp/IBM-controlled infrastructure and is triggered directly from Terraform workflows. The agent's implementation is not exposed to customers. Customers interact with it through Terraform and receive its outputs.

### 2. Customer use cases

**Core six (directly scoped to AGX agent builds):**

| Use case | What it does |
|---|---|
| Module Upgrades | Identify version changes, assess downstream impact, propose code updates, guide validation |
| Terraform Version Upgrades | Detect compatibility issues, highlight deprecated patterns, recommend upgrade changes |
| Terraform Provider Upgrades | Analyze breaking changes, identify affected resources, suggest config updates |
| Workspace Refactoring & Stack Migration | Restructure workspaces, split/consolidate configs, migrate to stacks-based patterns |
| Importing into Modules | Import existing infrastructure into reusable module structures |
| Chat to query Explorer | Natural-language questions about infrastructure, state, runs, policies, relationships |

**Five additional identified areas:**

| Use case | What it does |
|---|---|
| Unmanaged Inventory + Compliance Scan | Discover unmanaged resources, run compliance scans, optional remediation/import follow-on |
| Policy Issue Resolution | Explain policy failure, identify violation cause, provide remediation steps |
| Terraform Plan/Error Analyzer | Interpret plans, run failures, error output; explain root cause; prioritize fixes |
| Import - Optimized Config Generation | Generate cleaner production-ready Terraform config during import |
| Drift Remediation | Ingest drift events, explain cause, automate or recommend remediation |

### 4. 2026 areas of focus (priority order)

**Priority 1: TFE support**
- Current: AGX integrates with HCP-TF only.
- Desired: Full integration with customer self-hosted TFE environments.

**Priority 2: Multi-tenancy (in progress)**
- Current: Single tenancy only.
- Desired: Single tenancy for TFE, multi-tenancy for HCP-TF.

**Priority 3: Additional agent triggers**

| Trigger type | Current state | Desired state |
|---|---|---|
| Generic webhooks | Supported (requires config/customization) | - |
| Manual webhook | Not supported | Natively supported via UI button in HCP-TF and TFE |
| Run/Native Task webhooks | Not supported | Native support for run task and native task webhooks |
| Terraform Notification webhooks | Not supported | Native support for workspace and run events |
| Scheduled triggers | Not supported | Schedule-based execution (e.g. every 24 hours) |

**Priority 4: Connectors and integrations**

| Connector | Current state | Desired state |
|---|---|---|
| Terraform MCP | Static credentials only | Seamless authentication; native integration with remote MCP Server for HCP-TF agents |
| Terraform CLI + tfctl | Not supported | Seamless authentication for both |
| GitHub VCS writeback | Supported for HCP-TF | Needs TFE support for any config-writing use case |
| Terraform Skills | Not specified | Skills for writing HCL, building modules, writing policies |
| Customer-supplied Skills | Not supported | Allow customers to supply their own skills for customization |

**Priority 5: Agent orchestration - execution scalability**
- Current: Kubernetes-based, cluster scales with capacity.
- Desired: Handle potentially thousands of concurrent agent executions reliably.

### 5. Open decisions

**Bring-your-own provider vs. HashiCorp-managed provider**

This is the single most consequential unresolved product decision. Two models:

| Model | Description | Billing | Notes |
|---|---|---|---|
| HashiCorp-managed provider | HashiCorp supplies the model via AWS Bedrock; customer uses it | HashiCorp must pass token costs to customer; token limits needed | Current state; full control but billing complexity |
| Bring-your-own provider | Customer supplies their own provider keys (Bedrock, Azure Foundry, Vertex, Anthropic, OpenAI) | Customer billed by their own provider; no HashiCorp billing concern | Spacelift's current model; requires model validation per provider |

The memo notes that the hybrid model may be the right long-term direction but the immediate decision is which to prioritize for initial launch.

**Model limitations in BYOP model:** HashiCorp will need to restrict support to a defined subset of models per provider and validate each against every agentic workflow to confirm compatibility.

**Embedded small language model:** Referenced in the document with no content - appears to be a placeholder for future definition.
