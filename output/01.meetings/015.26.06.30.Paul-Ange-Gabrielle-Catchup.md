### 4. Nicole's explore chat

Nicole is a designer who was directed to build a global chat AI feature across Terraform, then narrowed to Explorer. She has been aggressive about it for two weeks. Her mocks and work are described as "reminiscent of the pattern we want to pull from Meridian into Terraform."

Paul's concern: forking engineering resources into a separate chat initiative that won't align is the real risk. If product management is directing her, that's a PM alignment problem, not a design problem.

Resolution path: Gabrielle and Ange to bring Nicole into the broader discussion; her research folds into the Meridian integration work, not runs parallel to it.

### 5. Run fixer first

Paul explicitly: "run fixer is a gateway into what we want to do as part of the big picture. It's a very simple but effective way to approach our problem and integrate agent-supported remediation. So I would say we do run fixer first."

This aligns with:
- Strategy doc 002 CUJ-01 (failed-run repair as the first vertical slice)
- Omar's level-one-first argument (004 meeting notes)
- The workshop brief's MLE framing

**This is the clearest PM confirmation of the failed-run repair priority in the pipeline to date.**

### 6. Events architecture concept

Paul had been building an events SDK - essentially killed in favor of the chat initiative - but the concept remains valid. The idea: all meaningful things happening across the Terraform app (workspace created, run kicked off, variable added, GitHub webhook received) are events that could be inputs to agents.

Run fixer = the run event. Version upgrade agent = a button-click event. CLI-driven too. Inputs can come from anywhere.

In-context responsiveness: depending on which page the user is on, the agent experience is context-aware. Nicole's explore work and this events concept are the same idea from different angles. Paul: "That would just be highly intelligent."

Gabrielle's follow-on question: can all events be compiled into a log that can be hooked for a "God's eye view"?

Paul's answer: two logs exist -
- TFE: file-based, comprehensive, "just a line added in Rails code somewhere"
- TFC: UI log

Both can be hooked; potentially a one-liner in Rails. **This makes the events layer much more accessible than assumed.**

### 7. tfctl API coverage

Paul's position: tfctl should have 100% API coverage of everything the UI can do. Agents need to be able to do anything the UI can do. Destructive actions (delete) were previously excluded but are being added.

Governance pieces are required before giving agents full-speed destructive access. This is consistent with the approval gate design in strategy doc 002.

### 8. AGX consolidation (reiterated)

Paul referenced the "lots of different agents that people are working on" and the fragmented approach, reiterating the direction from the 06/29 session: the goal is one cohesive thing, with Meridian as the center.
