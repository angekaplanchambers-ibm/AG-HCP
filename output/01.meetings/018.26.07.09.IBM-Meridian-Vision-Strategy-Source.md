IBM Meridian
Vision, Strategy & Shared Responsibility Model

Meridian is an intent based, proactive, operating layer that makes AI operations for infrastructure practical at enterprise scale. Meridian manages and allows querying/reasoning of 'operating environments'; on-premises, sovereign core, cloud, and edge infrastructure.

# Vision

We are building toward a future where every change is planned, simulated, and impact-analyzed before production is touched, where humans and AI agents operate within the same governed guardrails provided by our Trust Ladder, and where every action is verifiable and auditable. Change cycles compress from months to minutes, and the platform grows smarter with every environment it runs in.

Meridian can query multiple infrastructure types in real time to answer questions related to configurations of that infrastructure (with plans to leverage Infragraph).

Meridian is the ACT in the Observe/Sense -> Reason -> Act paradigm. Meridian can accept intent from humans, applications, or agents and acts on that intent.

Meridian will be the core foundation of Terraform's Commercial engine to meet the realities of the agentic horizon.

# Strategy

Provide the foundational capabilities to be extended for each operating environment. Meridian is the foundational building block, extended for operating environment specific requirements through Terraform providers, Ansible Playbooks, and direct API, CLI and MCP interactions where needed.

Land on the most demanding operating environment first, then extend. IBM Z and LinuxONE are Meridian's first anchor, the most complex and regulated infrastructure in the enterprise. Proving Meridian here establishes the credibility and the platform foundation that extend naturally to adjacent architectures.

Build a reusable coordination layer (AI Harness and AI Reasoning Loop). At the core is a coordination engine that creates and orchestrates workflows, enforces policy and trust, and supports simulation-first change planning. Meridian is designed as a reusable layer that interfaces with other tools to act: Terraform, Ansible, and direct API calls. Meridian can act against hyperscaler clouds, sovereign core, and on-premises infrastructure and appliances, delivering consistent automation and governance across diverse infrastructures.

The future of Terraform Commercial. Meridian can accept an intent, and reason on the operation to undertake. Meridian will be the core foundation of conversational/intent based operations for Terraform Commercial. Meridian will be embedded into HCP Terraform and Terraform Enterprise (for Commercial use only).

Grow the platform through an open modular ecosystem. Meridian is built to operate against HashiCorp Terraform and RedHat Ansible, leveraging Vault for identity and access. Leveraging domain-aware connectors built by IBM portfolio teams and ISVs via the SDKs/Providers to expand coverage with each environment we manage. Z and LinuxONE first; Power, hybrid cloud, and ISV environments next.

Governance is a differentiator. Sovereign and hybrid by design, with cryptographically-verified identity, workflows with client assigned Trust Ladder levels, immutable audit trails for compliance. This directly addresses where regulation (DORA, GDPR, sovereign-cloud mandates) is heading and is the foundation the broader roadmap is built on.

Safety is a differentiator. Meridian can conduct change simulation, and this provides additional safety levers. Meridian can simulate changes, and determine if any of the following concerns would be impacted: Security, Durability, Availability. See the Trust Ladder.

Leverage Infragraph for dependencies. When ready, Meridian can leverage the data of Infragraph to understand relationships across infrastructure estates. This should include VMware, PaloAlto, Cisco, NetApp, Dell, CSPs, Z, Power, DSK8000, etc.

# What Meridian is not

Meridian is an intent based operator that accepts natural language and structured instructions from humans, configurations, agents, and via APIs.

- Meridian is not an event ingress and processing engine. We do not process telemetry or big data. We will likely defer that responsibility to IBM Concert.
- Meridian is not a Terraform provider software factory. While we did build this functionality, we will transition that technology over to the PAVE team.
- Meridian is not a Software Factory play. While it might use an internal software factory for it's own velocity, that is not something we will productize.

# Meridian and Terraform Commercial

Meridian is the foundational block for Terraform Commercial to provide intent and conversational execution. Meridian's engine will be embedded into Terraform Commercial and will provide functionality to:

- Meridian name: The Meridian name should fade away, and this now Terraform Commercial's intent engine.
- Process intent via: conversational interfaces, intent based Terraform configurations, CLI, and direct APIs. Integration points to be placed within key Terraform Commercial workflows.
- Leveraging Project Solis: Meridian will be integrated into Terraform Commercial UIs using Project Solis. We can skin/theme Solis in the Terraform UIs. This provides us a conversational interface pre-built. Integrate it using a micro-front-end design pattern.
- Intent based Terraform configurations: Allow practitioners to specify intent in Terraform configurations with a new 'intent' block and generate Terraform configurations using a modality similar to config driven import. Intent block, plan, new code generated. "Build me a VPC that is well architected and leverages our VPC modules". Meridian processes the intent in SaaS/Self Hosted and generates the configurations. Practitioners can do this via the Terraform CLI.
- Intent based Terraform CLI: Like 'terraform action' have 'terraform intent XYZ' that agents can call, and the CLI proxies that call to Meridian to help generate code and identify dependencies that exist out of the root configuration.
- Terraform Remediation Agent JTBD will be rebased into Meridian. We will take the learnings from the Remediation Agent and build them out leveraging Meridian. Meridian will be the AI Reasoning Loop and AI Harness.

# Shared Engineering Responsibility Model

Meridian can manage resources that span multiple IBM teams: Z Software, Power, Terraform Ecosystem, Terraform Provider for AWS, etc. Responsibility is divided by domain: each portfolio team helps Meridian Core engineering to build the initial GA and owns the software and support for the components in its domain, Post GA. We build together with the domain experts and then transition ownership back into the domain. This keeps deep hardware and environment expertise with the teams that own it, while Meridian provides the common coordination, policy, and trust layer that ties the domains together.
