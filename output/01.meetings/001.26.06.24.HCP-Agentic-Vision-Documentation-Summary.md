## Key context

**Customer signal:** Airbnb is moving from TFE to Spacelift. Their stated gaps:
1. Agentic workflows, not isolated primitives
2. Verification and guardrails around agent output
3. Module testing tied to registry and HCP Terraform workflows
4. Failed-run explanation and embedded debugging
5. Visible product motion - faster releases, faster fixes

**Working phrase:**
> HCP Terraform should be the trusted place where infrastructure intent becomes verified change.

---

## Product loop

```text
intent -> context -> plan -> code -> validation -> approval -> run -> explanation -> remediation -> audit
```

Each team should be able to point its work to one or more links in this chain.

---

## Competitive landscape (condensed)

The market is splitting into five motions, not converging on "AI chat for IaC":

1. **Agentic infrastructure platforms** - Spacelift Intent, Pulumi Neo: full NL-to-infra loops
2. **IaC control planes with AI** - Scalr, Atlantis: run summaries, MCP, PR comments, drift
3. **Developer coding agents** - GitHub Copilot, Claude Code, Cursor: own the coding surface, reach infra via MCP
4. **Cloud-provider copilots** - Azure Copilot, AWS Q: contextual resource help and IaC generation
5. **Domain substrates** - Geniez: connect general assistants to governed domain systems

**Sharpest signal:** Spacelift has a connected loop:
```text
intent -> generated IaC -> policy -> run -> explanation -> drift -> remediation -> audit
```

**Key competitors by concern level:**

| Competitor | Why it matters |
|---|---|
| Spacelift | Direct TFE replacement; Airbnb chose it; explicit Ask/Build modes, hosted MCP, module tests |
| Scalr | Drop-in TFC alternative; PR-native GitOps, slash commands, `/scalr ai-explain` |
| Pulumi Neo | "infrastructure engineer" framing with enterprise controls, approvals, audit |
| env zero / Firefly | Governance, drift, cost, remediation - longer product story |
| GitHub Copilot | Sets PR/branch/check artifact expectations for all agents |
| Geniez | Domain substrate pattern: external agents stay general, domain system keeps audit/permissions |

## HCP Terraform's existing advantage

Assets already in place:
- Workspaces, runs, plans, state lineage
- Policy checks (Sentinel/OPA)
- Variables, registry/module context
- VCS integration, approvals
- Audit expectations, enterprise permissions
- Terraform MCP Server (registry, workspace, run management, tool filtering, per-user tokens)

**Gap:** These are primitives. Competitors are wrapping similar primitives in modes, hosted setup, approvals, audit UX, and end-to-end workflows.

## Recommended first slice: failed-run repair

```text
failed run -> evidence-grounded explanation -> workbench fix -> PR -> speculative run -> policy gate -> approval -> audit
```

**Why:** High customer pain (Airbnb explicit ask), feasible with existing data, high trust story, directly competitive with Spacelift run AI and Scalr ai-explain.

**Target experience:**
- Run page entry point with inline explanation and evidence citations
- Mode indicator: explain / plan / prepare PR / operate
- Workbench: proposed fix, diff, speculative plan, policy result, approval controls
- GitHub PR with concise run summary, failed-check detail, suggested action, workbench link
- Audit record tied to run/workspace history

**First failure classes (in order):**
1. Missing required variable
2. Invalid module input
3. Provider version conflict
4. Policy failure with clear remediation
5. Drift-caused plan surprise

## Non-goals (explicit)

- No fully autonomous production apply as first proof
- No generic AI chatbot
- No full HCP Terraform UI/IA redesign as prerequisite
- No blank-prompt multi-cloud provisioning demo
- No replacing GitHub code review
- No replacing policy-as-code

---

## Cross-surface handoff rules

1. Start anywhere - terminal, Slack, GitHub, and HCP UI are all valid entry points
2. Verify in Terraform - plans, policy, state, and runs stay in the trust layer
3. Act in reviewable units - PRs and run actions stay visible
4. Preserve evidence - each surface links back to the same agent session/audit record
5. Escalate the surface when needed - inline for explanation, workbench for consequential change
6. External agents carry intent; HCP Terraform owns validation, permissions, and durable proof

---

## Prioritization order (from workshop doc)

1. Failed-run repair (High pain, High feasibility, High trust story)
2. Module-backed intent to IaC
3. Drift remediation
4. Policy failure repair
5. Workspace lifecycle automation
6. Autonomous apply (High pain, Low feasibility, Low trust - deferred)
