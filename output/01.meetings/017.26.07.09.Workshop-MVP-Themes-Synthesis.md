# 017.26.07.09.Workshop-MVP-Themes-Synthesis

## TL;DR

Eight MVP themes synthesized from Research Sprints 1-4 (February-June 2026) across multiple participant cohorts. Three themes have universal, five-star signal strength and are table stakes for any MVP: blast-radius visibility before changes (Theme 1), governance and human-in-the-loop (Theme 2), and upgrade lifecycle toil (Theme 5). Trust through transparency (Theme 7) is a cross-cutting requirement present in every sprint - not a feature to add later.

The most important single data point: Sprint 4 participants overwhelmingly preferred **Option B** autonomy - agent bumps version, runs plan, rewrites config, opens PR; human approves merge and apply. This is the same autonomy level as CUJ-01 (failed-run repair) in strategy doc 002. It validates the workbench + PR creation + human approval flow as the correct interaction model across use cases, not just failed-run repair.

Two themes that are currently absent or underweighted in strategy doc 002 need elevation: blast-radius visibility (Theme 1) and upgrade lifecycle toil (Theme 5). Both have stronger sprint signal than failed-run repair.

## Key Discussion Points

### 1. Theme 1 - Blast-Radius Visibility Before Any Change (★★★★★)

**Signal**: Participants across all four sprints cannot confidently understand the full downstream impact of a module, provider, or workspace version change before applying it. Named as the single most cited source of fear and manual toil.

**Representative quotes**:
- "Nothing in the plan said 'this rebuilds your cluster.' A reviewer skimming the MR sees a trivial version bump and misses that it freezes deploys across every prod region." (Sprint 4, James)
- "A major provider upgrade had an undocumented schema change that triggered unexpected resource recreation in a multi-tenant environment." (Sprint 4, Janak)
- "A minor version bump on the azurerm provider silently dropped a diagnostic setting during apply. We only caught it in a post-deployment CSPM review." (Sprint 4, Harish)

**MVP direction**: AI-powered blast-radius analyzer. Given a proposed change: surface all affected workspaces and consumers, expected plan diff with risk annotations, known breaking changes from provider changelogs, and recommended human review checkpoints.

**Pipeline flag**: This use case is not in strategy doc 002 as a primary CUJ. It has stronger sprint signal than failed-run repair and directly overlaps with the upgrade lifecycle (Theme 5). Needs elevation.

---

### 2. Theme 2 - Governance, Identity, and Human-in-the-Loop Are Non-Negotiable (★★★★★)

**Signal**: Every participant in every sprint drew the same hard line - AI can assist, analyze, draft, and propose, but production changes require explicit human approval, least-privilege identity scoping, and a full audit trail.

**Representative quotes**:
- "The danger isn't the agent doing something obviously destructive, it's removing the human pause in front of a change whose risk the plan understates." (Sprint 4, James)
- "Agents never get permanent keys. They use short-lived, just-in-time tokens via OIDC or Vault. InfoSec and our architecture teams set the guardrails." (Sprint 4, Janak)
- "We do not trust agents in changing the infrastructure codes and stuff. We are testing it in an isolated environment." (Sprint 4, Harish)

**MVP direction**: Every agentic action must be observable (tool trace, approval state, identity), scoped to least privilege (short-lived credentials, OIDC/Vault JIT), and require explicit human approval before touching production. The Trust Ladder (L1 → L4) should be surfaced in UX so customers can incrementally adopt higher autonomy. MCP emerged as the preferred integration protocol for governance consistency (Sprint 4: Antonin, Adama, Harish, Rajendra).

**Pipeline flag**: This is already the core trust model in strategy doc 002. The MCP governance scope panel, approval gate, and audit trail designs are validated by this theme. No changes needed to current scope - confirm and proceed.

---

### 3. Theme 3 - Intent-Driven Operations Reduce Toil Without Replacing Expertise (★★★★☆)

**Signal**: Participants are ready to state what they want and let AI figure out the Terraform steps - but only when the system explains its reasoning, references authoritative sources, and produces a reviewable plan.

**Representative quotes**:
- "I use it much more as a collaborative agent - significantly better at understanding the context of an entire repository, following templatized patterns." (Sprint 4, Adama, on Codex)
- "If we get just the recommendations without any explaining or an official documentation... we can't trust the AI's response." (Sprint 3, Atul)

**MVP direction**: Chat/intent interface translating natural language infrastructure goals into Terraform plan previews with cited reasoning. Start narrow: intent → plan → human-reviewed apply for 3-5 top workflows (upgrades, new workspace, drift remediation). Maps to AGX Chat roadmap item (Q1 2027).

---

### 4. Theme 4 - Drift, Failures, and Anomalies Need Proactive Detection - Not Reactive Discovery (★★★★☆)

**Signal**: Participants consistently manage drift, failed runs, and configuration anomalies reactively. They want AI to surface issues before they become incidents, with enough context to act.

**Representative quotes**:
- "AI can handle alert correlation, detect anomalies, suggest initial root causes... I still validate AI outputs, any complex edge cases." (Sprint 3, Mohammed Hassan)
- Agent automatically scans Terraform files against CIS/NIST controls, flags findings mapped to file paths and line numbers - "Nothing sits in a log somewhere waiting to be noticed." (Sprint 4, Harish)

**MVP direction**: Estate health view continuously surfacing drift, failed runs, policy violations, cost anomalies, and upgrade risks in a single pane. AI proposes remediation steps with blast-radius context. System suggests; human approves and applies. Maps to AGX Agent: Analyzer roadmap item (Q3 2026).

**Pipeline flag**: This is the Slack pulse and admin observability surface in strategy doc 002. Validated. The "single pane" framing from sprint participants is stronger language than the current strategy doc - consider updating the Slack pulse CUJ-03 description.

---

### 5. Theme 5 - Upgrade Lifecycle Is the #1 Source of Ongoing Toil (★★★★★)

**Signal**: Module, provider, and Terraform workspace version upgrades are near-universal pain across all four sprints - slow, manual, high-risk, and disproportionately time-consuming.

**Representative quotes**:
- "It takes 6 to 9 months to upgrade. I have 400 plus GitHub modules. That's a labor cost. It's a time cost." (Sprint 3, Lada)
- "You can't go through and update everything. You can't be doing that all the time. You need a lot of manpower." (Sprint 4, Joel)
- "I've privately worked on an agentic solution that checks for new provider versions, opens a PR, and makes the version bump itself. It would save real toil." (Sprint 4, Joel)

**Sprint 4 Day 3 survey result**: Majority preference is **Option B** - "Agent bumps the version, runs the plan, rewrites config, and opens a PR" - with Option C (auto-PRs for clean plans) as near-future goal.

**MVP direction**: AI-assisted upgrade workflow: (1) detect outdated providers/modules across all workspaces, (2) analyze breaking changes from changelogs, (3) map blast radius, (4) draft updated `.tf` files, (5) open PR with pre-computed plan for human review. Human approves merge and apply. Maps to AGX Agent: Upgrade roadmap item (Q4 2026).

**Pipeline flag**: This is CUJ-05 (module-backed intent-to-IaC) in strategy doc 002, currently classified as "supporting." Sprint signal is five stars - same as blast-radius visibility. Needs primary CUJ status. Customer key asks document (005) already flagged this; the sprint data confirms it emphatically.

---

### 6. Theme 6 - Self-Service Infrastructure Requires Golden Paths, Not Open Playgrounds (★★★☆☆)

**Signal**: App developers and less-experienced engineers need pre-approved standardized templates they can parameterize and deploy - reducing platform team bottlenecks while preserving governance.

**MVP direction**: Blueprint-based self-service (partially shipped as developer portal/blueprints). Key gaps: clean GitHub integration, AI assistance for filling blueprint parameters from natural language intent, plan preview before provisioning.

---

### 7. Theme 7 - AI Trust Is Earned Incrementally Through Transparency and Explainability (★★★★☆, cross-cutting)

**Signal**: Participants do not resist AI. They resist AI they cannot verify. Trust grows when AI shows its work, cites sources, admits uncertainty, and stays within defined boundaries.

**Representative quotes**:
- "If we get just the recommendations without any explaining or an official documentation... we can't trust the AI's response." (Sprint 3, Atul)
- "Traditional monitoring tells us what happened, but not always why a particular action was taken, what context was used, or whether the decision aligned with the intended policy." (Sprint 4, Antonin)
- "I still validate plans and review changes... I do trust it more than I did a year ago because of these enhancements, but it's blind trust." (Sprint 4, Adama)

**MVP direction**: Every AI-proposed action must include what it will do and why, what it consulted (changelogs, provider docs, policy), what it is uncertain about, and a human-reviewable plan before execution. Confidence levels must be surfaced. System must flag when operating at the edge of its knowledge. Table stakes for regulated-environment adoption (BFSI, healthcare, government customers across all four sprints).

**Pipeline flag**: This is the evidence panel, confidence label, and uncertainty notes in the CUJ-01 journey map (005). Validated. The confidence label and "uncertainty notes" design decisions are not optional - they are the mechanism by which regulated customers adopt at all.

---

### 8. Theme 8 - Onboarding and Time-to-Value Remain Barriers (★★★☆☆)

**Signal**: New Terraform adopters and teams expanding usage face steep learning curves around module selection, import/migration, policy setup, and workspace organization. AI has not meaningfully accelerated this path yet.

**MVP direction**: AI-guided onboarding for new workspace creation, import of unmanaged resources, and first-run module selection. Maps to AGX Agent: Import roadmap item (Q1 2027).

---

### Cross-cutting: The Option B consensus

Sprint 4 data resolves a key design question. Across all upgrade-class workflows, participants overwhelmingly chose:

> **Option B**: Agent bumps the version, runs the plan, rewrites config, and opens a PR. Human approves merge and apply.

Not Option A (human does it manually with AI suggestions) and not Option C (agent auto-closes if plan is clean). This is the exact interaction model in CUJ-01's journey map - workbench + speculative run + PR creation + human approval. The sprint data confirms this is the right autonomy level for the MVP across multiple use cases, not just failed-run repair.
