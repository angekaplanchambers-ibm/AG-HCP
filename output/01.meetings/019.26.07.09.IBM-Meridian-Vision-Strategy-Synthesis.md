# 019.26.07.09.IBM-Meridian-Vision-Strategy-Synthesis

## TL;DR

This document is the primary source of record for Meridian's intent and direction as an IBM platform product. Four signals are directly relevant to the HCP Terraform agentic design pipeline:

1. **The Meridian name will fade.** Meridian becomes Terraform Commercial's embedded intent engine. This is the strategic direction, not a speculation.
2. **Project Solis is the pre-built conversational UI** that Meridian will integrate into Terraform Commercial UIs as a micro-front-end. This is the implementation path for the workbench and chat interfaces in our wireframe plan.
3. **Terraform Remediation Agent JTBD is explicitly rebased into Meridian.** CUJ-01 (failed-run repair) is confirmed as Meridian's scope, not a parallel effort. Meridian will be the AI Reasoning Loop and AI Harness for this use case.
4. **Trust Ladder is named explicitly** as the governance mechanism for Meridian workflows. Our PDR-001 Trust Ladder model is directly aligned with Meridian's stated governance differentiator.

One signal that is not yet reconciled: this document describes Meridian's "AI Harness and AI Reasoning Loop" as the coordination engine. In our pipeline, that role is occupied by AGX (Marc Cosentino's backend). Whether Meridian's AI Harness and AGX are the same system, converging systems, or parallel systems is a question for Chris Audie and Daniel Savage. Do not update architecture docs until this is confirmed.

## Key Discussion Points

### 1. Meridian's role: the ACT layer

Meridian is positioned as the ACT layer in an Observe/Sense → Reason → Act paradigm. It accepts intent from humans, applications, or agents and acts on that intent. It is not a monitoring system, not a telemetry processor (deferred to IBM Concert), not a provider factory (transitioning to the PAVE team). It is the execution and coordination layer.

This is a clean scope boundary that aligns with our layer separation in strategy doc `003`: AGX/Meridian = backend execution; HCP Terraform UI = frontend surface. Meridian's stated scope matches what our architecture expects from the backend layer.

### 2. Meridian name will fade into Terraform Commercial

The document states explicitly: "The Meridian name should fade away, and this is now Terraform Commercial's intent engine."

This resolves the strategic ambiguity from Paul's earlier comments ("AGX will probably mostly go away"). The current picture is: Meridian's engine is the intent processing and execution layer; it will be embedded into Terraform Commercial under the Terraform brand; the Meridian name is a transitional label. This is not speculation - it is written as the current strategic direction in this doc.

Implication for our pipeline: the design and UX work we are doing for the HCP Terraform agentic surfaces IS Meridian's Terraform Commercial integration work, even if it is not currently labeled as such. The workbench, run page entry, Slack pulse, and audit view are the surfaces Meridian will operate through.

### 3. Project Solis - the pre-built conversational UI

New signal: Meridian will be integrated into Terraform Commercial UIs using **Project Solis**. Solis provides a conversational interface pre-built. The integration pattern is micro-front-end. The team can skin/theme Solis in the Terraform UIs.

This is directly relevant to PDR-004 and our workbench/chat entry design. Project Solis is likely what the Meridian PRD refers to when it describes the MER-16 chat interface. Before finalizing workbench wireframes, we need to understand what Solis provides out of the box so we are not designing something Solis already ships.

Action needed: Identify who owns Project Solis and what it delivers. Add to open questions for Chris Audie or Daniel Savage.

### 4. Intent block in Terraform configurations

New interaction modality not in our strategy docs: an `intent` block in Terraform HCL configurations, working like `config-driven import`. The practitioner writes an intent statement in their `.tf` file; Meridian processes it and generates the Terraform configuration.

Example from the doc: `"Build me a VPC that is well architected and leverages our VPC modules"` → generates the HCL.

This is a distinct entry point from the run page and workbench in our current CUJ design. It is a CLI/IaC-native interface rather than a UI-native one. Maps loosely to CUJ-05 (module-backed intent to IaC) but through a different surface.

### 5. `terraform intent XYZ` CLI command

A new CLI modality: `terraform intent XYZ` that agents can call. The CLI proxies the intent to Meridian to generate code and identify dependencies outside the root configuration.

This is the agentic CLI entry point (CUJ-04 surface variant in our architecture). Agents calling `terraform intent` rather than MCP tools. Not in our current PDR-001 surface inventory - would need to be added if this CLI command is coming in the six-month window.

### 6. Terraform Remediation Agent → Meridian

The document explicitly states: "Terraform Remediation Agent JTBD will be rebased into Meridian. We will take the learnings from the Remediation Agent and build them out leveraging Meridian. Meridian will be the AI Reasoning Loop and AI Harness."

This confirms CUJ-01 (failed-run repair) is Meridian's domain, not a separate effort. The Remediation Agent work (which appears to be prior art or a parallel workstream) is being absorbed into Meridian. This is the clearest signal yet that the design work in our PDR chain is aligned with where Meridian is heading.

### 7. AI Harness and AI Reasoning Loop - naming vs. AGX

The document names the Meridian core as the "AI Harness and AI Reasoning Loop." These terms do not appear in our AGX documentation (which uses "AGX" as the label). Two possibilities:

- **Same system**: AGX = Meridian's AI Harness/Reasoning Loop (different team naming for the same capability)
- **Different systems**: AGX is a separate IBM/HashiCorp backend; Meridian has its own AI Harness

If AGX and Meridian's AI Harness are converging, the "AGX will mostly go away" signal from Paul aligns with this document. If they are separate, there are two backend runtime systems competing for the same frontend surfaces.

This must be resolved before PDR-001's trigger/response architecture is finalized. Do not act on this until confirmed.

### 8. Trust Ladder is named as a Meridian governance primitive

The document names Trust Ladder as a first-class governance mechanism: "workflows with client assigned Trust Ladder levels." This validates our PDR-001 Trust Ladder model (L1-L4) as the correct governance abstraction. It is not just a design metaphor - it is named in the Meridian strategy doc as a product primitive.

### 9. Governance and safety as differentiators

The doc names two explicit differentiators for Meridian:
- **Governance**: cryptographically-verified identity, immutable audit trails, DORA/GDPR compliance
- **Safety**: change simulation before production; impact on Security, Durability, Availability

Both are directly implemented in our design pipeline: the audit trail (Stage 9 in PDR-002/003), the speculative run validation (Stage 6), and the approval gate (PDR-001) all map to these differentiators.

### 10. Shared responsibility model

Meridian Core owns the platform (AI Harness, Reasoning Engine, coordination engine, agents, workflow catalog). Domain teams (Z Software, Power, Storage, CSPs) own their providers/components post-GA. The responsibility table in the source doc is partially populated for Z Stack; other environments (Power, AWS, Azure, GCP, OnPrem) are listed without detail.

For HCP Terraform design work, the implication is that Meridian Core is the engineering counterpart for the intent engine surfaces, while Terraform Ecosystem owns the HCP TF-specific integration components.

### 11. Infragraph - future dependency, not MVP

The doc mentions Infragraph as a future capability for understanding relationships across infrastructure estates. This would enable blast-radius analysis (CUJ-06) with real dependency graph data. It is explicitly scoped as "when ready" - not in the current window. Our CUJ-06 blast-radius design should not depend on Infragraph being available.

### 12. What Meridian is not (scope exclusions)

- Not an event ingress/telemetry processor (IBM Concert handles that)
- Not a Terraform provider factory (transitioning to PAVE team)
- Not a software factory

For design scope: we should not design telemetry dashboards or provider management surfaces as Meridian features. The admin observability dashboard in our scope backlog (strategy doc 002) is a Meridian/AGX session trace surface, not a telemetry processing surface.
