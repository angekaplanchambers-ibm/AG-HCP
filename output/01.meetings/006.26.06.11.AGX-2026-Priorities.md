# Vision
The AGX team's objective is to provide a platform to solve real-world customer problems via Agents and Agentic solutions.
AGX’s vision is to be IPG’s central platform for Managed Agents, enabling teams to quickly build, govern, and embed agentic capabilities into real product workflows that automate high-value customer operations. By providing a flexible, reusable foundation for agent orchestration, security, policy, and observability, AGX will accelerate delivery of trusted agents that improve customer outcomes while reducing operational toil. The platform will support both Terraform Enterprise and HCP Terraform (HCP-TF), ensuring a consistent managed-agent experience across self-managed and cloud environments.
# Scope
## Customer Use Cases:
To ensure we have desired line of sight into what needs to be built into the platform, we have identified the following are a core set of targeted customer agentic workflows that the team is looking to solve (from the Terraform Agentic Shaped Customer problems document). Each of these customer scenarios will result in building an agent within the AGX platform and hooking it up to a workflow with HCP-TF and TFE.
Module Upgrades: Help customers safely upgrade Terraform modules by identifying version changes, assessing downstream impact, proposing code updates, and guiding validation before rollout.
Terraform Version Upgrades: Assist teams in moving between Terraform versions by detecting compatibility issues, highlighting deprecated patterns, and recommending the changes needed for a smooth upgrade.
Terraform Provider Upgrades: Reduce the effort of provider upgrades by analyzing breaking changes, identifying affected resources and arguments, and suggesting configuration updates and validation steps.
Workspace Refactoring & Stack Migration: Support large-scale infrastructure reorganization by helping customers restructure workspaces, split or consolidate configurations, and migrate to stacks-based patterns with less manual effort and risk.
Importing into Modules: Simplify adoption of Terraform best practices by helping customers import existing infrastructure directly into reusable module structures instead of flat, hand-built configurations.
Chat to query Explorer: Enable users to ask natural-language questions about infrastructure, state, runs, policies, and relationships so they can quickly understand their environment without manually digging through multiple screens or outputs.
Plus, additional customer areas have been identified:
Unmanaged Inventory + Compliance Scan: Leverage Terraform Search together with Terraform Policy to discover unmanaged resources and run compliance scans across workflows ranging from targeted queries to full account scans, with optional follow-on agent support focused on remediation, config fixes, or optimized import generation rather than the scan itself.
Policy Issue Resolution: Help users understand why a policy failed, what configuration caused the violation, and what concrete remediation steps are needed to achieve compliance of the policy violation.
Terraform Plan/Error Analyzer: Interpret plans, run failures, and error output to explain root cause (ie. Misconfiguration, policy violation, etc), prioritize likely fixes, and reduce time spent troubleshooting Terraform workflows.
Import – Optimized Config Generation: Generate cleaner, more production-ready Terraform configuration during import by reducing noisy output and aligning generated code to preferred patterns and module structures.
Drift Remediation: Ingest occurrences from drift detection, explain how the environment drift to help customers decide the best course of action and automate desired remediation actions.

## AGX Platform Components
The following are key components that have been identified as necessary to be a part of the platform to support the customer use cases above. Note, these are not in priority order, nor are all in scope for 2026 (see the Areas of Focus for 2026 section for 2026 prioritization).
HCP-TF and TFE Integration: The platform should integrate natively with both HCP Terraform and Terraform Enterprise so agents can operate consistently across cloud and self-managed environments.
Trigger Framework: The platform should support multiple ways to invoke agents based on product workflows and operational events. Triggers should include webhooks (ie. run tasks or native tasks), notification APIs, user-initiated actions, and schedule-based execution for proactive or recurring jobs
Connector and authentication layer: Agents need secure access to the systems they operate on. The platform should provide built-in connectors, credential handling, and policy-based authentication for Terraform MCP, Terraform CLI, TFE CLI (tfctl), GitHub and future enterprise integrations such as customer supplied MCP’s that would be relevant to their own environments.
Skills: The platform should integrate with skills to provide agents with domain-specific actions, tools, prompts, and guardrails into composable units. This allows teams to standardize common capabilities such as upgrade analysis, policy remediation, import generation, and workspace inspection.
Agent orchestration: The platform should manage multi-step workflows that involve planning, tool use, state transitions, retries, approvals, and result reporting. Orchestration should make agent behavior deterministic enough for enterprise workflows while still allowing flexible reasoning and tool selection.
Multi-agent coordination and handoff: The platform should support coordination between a primary agent and specialized sub-agents for tasks such as analysis, code generation, policy review, and validation. Handoffs should preserve context, execution history, and ownership boundaries so complex workflows can be decomposed safely.
Execution scalability: For managed Agents, scale is a huge concern. The platform should scale potentially thousands (exact number to be confirmed) of concurrent agent executions without degrading reliability. This includes horizontal scaling of workers, bounded concurrency controls, isolation across tenants, and backpressure mechanisms during bursts.
Queueing and job control: The platform should provide durable queueing for agent executions, so work can be prioritized, delayed, retried, paused, resumed, or cancelled. This is especially important for long-running tasks, scheduled operations, and workflows waiting on external systems or human approval.
Agentic SDK: Internal teams should have a simple SDK and developer framework for defining agents, sub-agents, triggers, connectors, policies, and execution of workflows. The SDK should reduce custom wiring, enforce platform conventions, and make it easy to onboard new use cases quickly.
Model Selection: The platform should include a model-selection layer that allows teams to choose the right model (ie. Anthropic Claude, OpenAI GPT, etc) for a task based on latency, cost, capability, enterprise policy, and workload type. This allows Agent development teams to optimize speed on simple tasks and depth or reliability on higher-value workflows.
Bring your own model Provider support: Customers should be able to supply their own provider keys or route through their approved model endpoints (with their own API keys). Support should cover AWS Bedrock, Azure Foundry, Google Vertex, Anthropic, and OpenAI, while preserving tenant isolation, auditability, and policy enforcement.
Context, memory, and RAG services: Agents need access to relevant operational and product context to solve complex workflows accurately. The platform should provide memory and retrieval services that can supply workspace metadata, prior run history, policy context, documentation, and customer-specific knowledge at execution time.
Vector and contextual data storage: The platform should provide storage for embeddings and retrieved context, along with lifecycle controls for freshness, retention, and tenant isolation. This should support both shared platform knowledge and customer-scoped contextual data.
Execution environments: The platform should support well-defined execution environments for different classes of agent work, including analysis-only tasks, Terraform-aware workflows, and tool-augmented remediation flows. Environments should be reproducible, auditable, and configurable per use case.
Sandboxing and credential isolation: Agents should be executable in sandboxed environments with tightly scoped credentials and explicit access boundaries. This is required to reduce blast radius, support least-privilege access, and safely enable write-capable workflows such as remediation or upgrade automation.
AGX Bridge:
The AGX Bridge coordinates identity and access across agent workflows by securely delegating permissions and managing identity handoffs between agents as work moves from one agent to another.
Billing and metering: The platform should meter token usage, model calls, tool execution, and infrastructure consumption so usage can be allocated, reported, and eventually billed. Metering should support internal chargeback as well as future customer-facing pricing models.

## Areas of Focus for 2026
The following are the focus areas for 2026 for the AGX Platform and are in priority order:
Provide Support for TFE
Current State: Currently, the AGX platform is integrated with the HCP-TF platform.
Desired State: To ensure we can support all our customers for the customer use cases identified above, the AGX platform should be able to be integrated with customers' self-hosted TFE environments.
Multi-tenancy for HCP-TF & single tenancy for TFE: (in Progress)
Current state: The AGX Platform currently supports single tenancy.
Desired state: The AGX Platform needs to be able to support single tenancy for TFE and multi-tenancy for HCP-TF. Note: This is currently in progress.
Support for additional Agent Triggers:
Current Supported Triggers:
Generic Webhooks – The AGX Platform currently supports generic webhooks which require configuration and customization.
Desired supported Triggers:
Manually triggered webhook – The AGX platform should natively support manually triggered webhooks. This would be used to support agentic workflows initiated by a user initiated via a button within the HCP-TF and TFE UI.
Run/Native Task Webhooks – The AGX Platform should natively support Terraform run/native task webhooks. This will ensure a seamless trigger from run task or native task webhooks to support agentic workflows that need to get triggered during a run.
Terraform Notification Webhooks – The AGX Platform should have native support for triggers from workspace events & run events. This could be used for use cases such as building an agentic workflow to analyze the output and error states of a plan.
Scheduled triggers – To support use cases such as finding unmanaged infrastructure or periodic Policy checks on currently managed infrastructure, the AGX Platform should allow for Agents to be triggered on a set schedule (ie. Every 24 hours).
Connectors and integrations:
Current Connectors:
MCP – Authentication for MCP currently utilizes static credentials
Github – The AGX Platform is currently integrated with GitHub and has VCS writeback capabilities (for HCP-TF environments).
Desired Connectors:
Authenticated Terraform MCP – The AGX Platform should support seamless authentication between agent and the Terraform MCP
Native integration with the remote Terraform MCP Server for agents operating in HCP-TF.
Terraform CLI & tfctl – The AGX Platform should support seamless authentication between agent and the Terraform CLI & tfctl
The AGX Platform needs to support integration with GitHub & the VCS writeback for TFE. This will be used to support any use case that involves writing Terraform configuration (ie. Upgrades, Policy error resolution, etc).
Skills – The AGX Platform should support the following flavors of Skills.
Terraform Skills – Managed Agents that are housed in the AGX Platform will need to leverage Terraform Skills to be able to know how to handle specific operations (ie. Writing HCL configuration, building modules, writing policies, etc)
Customer supplied Skills – The AGX Platform should allow customers to have some customization/control over the output of the agent, therefore supplying their own Skills may be required (ie. A skill to inform the agent how a module should be developed).
Agent Orchestration:
Current Orchestration capabilities:
Multi-Agent Handoff: The AGX Platform currently support multi-agent workflows, allowing Agents and sub-Agents to work together and hand off tasks between themselves.
Execution Scalability: The AGX Platform is currently based on Kubernetes, which allows the platform to scale the cluster based on the capacity required.
Desired Orchestration capabilities:
Execution Scalability: The AGX Platform will need to handle the scalability demands of potentially thousands (number needs to be determined) of concurrent agent executions.
## Open Decisions:
Q: Do we allow our customers to bring their own Provider or supply a Provider for customers?
HashiCorp needs to decide how it wants to bring AI solutions to market: should the product offer a HashiCorp-managed provider and model as part of a fully integrated experience, or should it allow customers to bring their own providers? This choice will directly shape the product strategy, including the core value proposition, the level of customer control and flexibility, the billing and ownership model, the scope of validation and support, and the product’s appeal to enterprise buyers. While a hybrid model may be the right long-term direction, the immediate product decision is which approach to prioritize for the initial launch.
Provider/Model Selection:
Current State: Currently development teams can choose a model that is available within the AWS Bedrock Provider when they are building agents in the AGX Platform – Note: Model availability is currently constrained by the set of Bedrock models enabled in our AWS account (ie. Claude, GPT, Nova, etc).
Desired State: To support this model we will need to introduce Billing to pass AI costs to customers. Desired Billing capabilities:
The AGX Platform will need the ability to pass the token costs incurred in our AWS Bedrock account to customers. Note: more requirements will need to be determined to figure out the exact billing mechanism and Pricing and Packaging for customers.
The AGX Platform will need to implement token limits for customers who are using agents via our AWS Bedrock Provider as unexpected costs could potentially be incurred by the customer.
Bring your own Provider support
Current State: This is currently not supported by the AGX Platform as currently the platform is constrained to models available within our own Provider (AWS Bedrock).
Desired State: Customers should be able to plug their own model Providers into the AGX platform (AWS Bedrock, Azure Foundry, Anthropic, OpenAI Google, etc). This becomes more important as customers will want to plug their own Providers into the agents to ensure control and privacy is accounted for when using AI for their workloads. In this model, billing is no longer a concern as the customer will be billed by their own providers for their AI usage. Note: This is the paradigm that SpaceLift has adopted for their AI workloads.
Model limitations: In a bring-your-own-provider model, the AGX platform will need to restrict support to a defined subset of models within each provider to ensure agents behave reliably and produce consistent outcomes. This will also require HashiCorp to validate multiple models against each agentic workflow to confirm compatibility and verify that the desired results are achieved.

## Definitions:
Managed Agent: A managed agent is an AI agent that is designed, hosted, operated, and maintained by HashiCorp/IBM as part of the Terraform product experience. It runs within HashiCorp/IBM controlled infrastructure and is triggered directly from Terraform workflows, such as through product actions (ie. Webhooks), UI controls, or workflow events like a plan or policy check. The agent’s implementation is not exposed to customers. Customers will interact with it through Terraform and receive its outputs, such as proposed Terraform configuration changes, remediation guidance, or analysis of policy and workflow failures.
Tools: A tool refers to a construct in MCP. In practice there are three common patterns:
MCP tool wrapping an existing API. The tool validates model input, calls your REST or GraphQL API, then returns the result.
MCP tool as a thin adapter over internal code. There may be no external API at all. The handler directly runs application logic, database queries, or filesystem operations.
MCP tool as an orchestration layer. The tool may call several APIs, merge results, and present one model-friendly output.
Provider: A provider is the platform or service that makes AI models available for use within a product. It handles access to the models, commercial terms, authentication, usage controls, and often the surrounding operational and governance capabilities. Examples of providers include AWS Bedrock, OpenAI, Anthropic, Google, and Azure AI Foundry.
Model: A model is the specific AI system used to generate responses, reason over inputs, or perform tasks inside the product. Different models vary in capability, performance, cost, latency, and behavior, which makes model selection an important product and technical decision. Examples of models include Claude, GPT, Nova, and Gemini.
