# 005. User Journey Map -- CUJ-01: Failed-Run Repair from Run Page

**Date**: 2026-07-01
**Author**: PPD synthesis
**Status**: Draft
**Depends on**: [002.26.06.24.HCP-Agentic-Vision-Strategy]
**Grounded in**: [03-vertical-slice-failed-run-repair.md], [002.26.06.24.HCP-Agentic-Vision-Strategy], [003.26.07.01.AGX-Meridian-Frontend-Backend-Synthesis], [004.26.07.01.AGX-Priorities-Meridian-Readiness-Map], [007.26.07.01.AGX-2026-Priorities-Parse], [015.26.06.30.Paul-Ange-Gabrielle-Catchup]

**Scope**: A run fails in HCP Terraform. The platform engineer or app developer triggers the agent from the run page, receives an evidence-backed diagnosis, reviews a proposed fix in the workbench, validates it with a speculative run, approves PR creation, and completes the workflow with a full audit trail.

---

## Personas

| Persona | Role | Category | Mental Model |
|---|---|---|---|
| Platform engineer | Owns workspace estate, on-call for production failures | Primary user | "Show me the evidence, then I'll decide what to do" |
| App developer | Writes HCL, opens PRs, depends on shared modules | Secondary user | "I don't know Terraform internals - tell me what went wrong and what to change" |
| Terraform admin (audit) | Reviews agent actions for compliance | Reviewer | "What did the agent read and propose, and who approved it?" |

### Platform engineer

**Category**: Primary user
**Quote**: "I can tell a run failed. I can't always tell why in under ten minutes, especially when a module or variable set is involved." [Strategy doc 002]

**Context**:
- **Role**: Manages dozens to hundreds of workspaces; on-call for production failures; owns run health and policy compliance
- **Environment**: HCP Terraform run page, Slack notifications, GitHub PRs, occasional terminal
- **Frequency**: Multiple failed runs per week in large estates; on-call rotations mean failures arrive at any hour

**Goals**:
- Understand root cause of failure without manually scanning the plan log
- Get a proposed fix they can verify before touching production
- Create a PR that has enough context to be reviewed without reopening the run

**Pain Points**:
- TFE/TFC does not explain failures - raw log only
- No way to trace failure to variable set or module source without deep knowledge of the workspace
- No evidence chain connecting the proposed fix back to the original failure

**Design Implications**:
- Explanation must cite specific evidence: exact log line, plan section, policy result, variable name
- Confidence label required - the agent should not present a guess as a diagnosis
- Workbench must show what the agent read and why it proposed the fix

### App developer

**Category**: Secondary user
**Quote**: "Speculative plan failed. The error references a module I didn't write. I have no idea which variable is wrong." [Strategy doc 002]

**Context**:
- **Role**: Writes HCL for their service's infra; depends on private registry modules; does not own the workspace
- **Environment**: GitHub PR review, HCP Terraform run page linked from PR
- **Frequency**: PR-triggered speculative failures during active development cycles

**Goals**:
- Understand the failure without needing platform team escalation
- Get a suggested fix fast enough to unblock the PR
- Trust that the agent-proposed change will not break policy

**Pain Points**:
- GitHub PR comment says "failed" with no diagnosis
- Module errors reference internals they do not have access to read
- Every non-trivial failure requires asking the platform team

**Design Implications**:
- PR comment must be self-contained and actionable without opening HCP Terraform
- Module source and variable context must surface in plain language
- Mode indicator must be visible so the developer knows the agent is proposing, not applying

---

## Jobs To Be Done

When a run fails in a workspace I own or depend on,
I want to understand the root cause and get a reviewable fix without manual log triage,
So that I can unblock the team, preserve audit accountability, and avoid introducing a second failure.

- **Functional**: classify the failure, cite evidence, propose a minimal safe fix, validate it speculatively, create a reviewable PR
- **Emotional**: feel confident in the proposed fix before touching production; feel in control of the agent's scope
- **Social**: produce an audit record the security team accepts; create a PR a reviewer can approve without reopening the run

---

## Critical User Journeys

**Primary**: As a platform engineer, I want to understand why my run failed and get a validated fix in a reviewable PR so that I can unblock the workspace without manual log triage or a second failure risk.

**Supporting**: As an app developer, I want the GitHub PR comment to tell me exactly what failed and what to change so that I can resolve the speculative run without escalating to the platform team.

## Per-Stage Detail

### Stage 1: Trigger

**User Action**: Platform engineer lands on a failed run page (via direct navigation, email notification, or Slack pulse). App developer sees a failed speculative run in a GitHub PR comment.

**Agent Action**: None in this stage. The run has already completed and failed. Run data is available in HCP Terraform.

**Touchpoint**: Run page (primary). GitHub PR comment (secondary entry for app developer, leads to Stage 2 via workbench link).

**System Calls**: None new. Run data, plan log, policy results, and workspace metadata are already stored.

**Data In**:
- Run ID, workspace ID, run status = failed
- Plan log (raw)
- Policy result (pass/fail with policy set context)
- Workspace variable set, VCS repo reference, module sources used
- Last successful run reference

**Data Out**: None. This stage is observation only.

**Validation**: None.

**Failure Mode**: User is not notified of the failure promptly (Slack pulse not yet triggered - depends on AGX notification webhook, desired state). User navigates to the run page anyway.

**Duration**: 0 seconds (passive event). User arrival at run page: variable.

---

### Stage 2: Request

**User Action**: User clicks "Ask Terraform" or "Explain this failure" on the run page. This is an explicit opt-in - the agent does not run automatically in this stage.

**Agent Action**: Trigger sent to AGX via manual webhook. AGX receives the run ID, workspace ID, and user identity. Agent begins context assembly.

**Touchpoint**: Run page entry point - inline button, visible on the failed run page header or adjacent to the error summary.

**System Calls**:
- UI → AGX: manual webhook trigger [AGX Priority 3, desired state - currently blocks production; design proceeds against mock]
- AGX → Terraform MCP Server: read run data, plan log, policy results, workspace context

**Data In**:
- User identity (token passthrough via Terraform MCP)
- Run ID, workspace ID
- Agent mode selection (default: explain; user may be able to select explain / plan / prepare PR)

**Data Out**:
- Trigger event with run context to AGX
- Mode indicator becomes visible to user: "Agent is working - explain mode"

**Validation**: User identity verified via MCP token passthrough. Run must be in failed state to trigger (guard: do not trigger on passing runs).

**Failure Mode**:
- Manual webhook not yet available (AGX Priority 3 gate) - button exists but calls a mock; agent response is simulated
- User clicks button but connection to AGX fails silently - UI must surface a failure state, not spin indefinitely
- MCP token is expired or lacks read scope - error surfaced in MCP scope panel before trigger fires

**Duration**: Button click → acknowledgment: under 1 second. Button click → first agent response: target under 10 seconds [A, from Meridian PRD MER-16].

---

### Stage 3: Diagnosis

**User Action**: User reads the inline explanation panel that appears below or adjacent to the run error summary. No action required - this is a read interaction.

**Agent Action**: Agent classifies the failure against the defined failure classes (missing required variable, invalid module input, provider version conflict, policy failure with clear remediation, drift-caused plan surprise). Cites specific evidence. Returns structured diagnosis.

**Touchpoint**: Inline explanation panel on the run page. Expands in place; does not navigate away.

**System Calls**:
- AGX → Terraform MCP Server: read plan log (specific error lines), policy result (failing policy name, rule, cited resource), workspace variable set (which variables are set, which are missing), module source (registry path, version), last successful run (for drift comparison)

**Data In** (agent reads):
- Plan log - specific error section, not the full log
- Policy results - failing policy name, rule text, resource cited
- Workspace variable set - variable names, whether required variables are absent
- Module source - registry path, pinned version, whether version constraint is broken
- Run history - last successful run timestamp and SHA

**Data Out** (inline panel shows):
- Failure classification (one of the five defined classes)
- Evidence citations: specific log line number, plan section, policy rule name, variable name, module reference
- Workspace context: which variable set applies, which module version is in use
- Confidence label: high / medium / low with brief rationale
- Uncertainty note: what the agent could not determine (e.g., "could not access external module source")
- Agent mode indicator: "Explain mode - no changes proposed yet"

**Validation**: Every evidence citation must reference a real artifact (log line, plan line, policy rule) that exists in the run data. Agent must not produce generic text that is not grounded in the specific run.

**Failure Mode**:
- Agent produces generic explanation not tied to actual log/plan/policy evidence - confidence label must be "low" and uncertainty note must state what was not found
- Failure class does not match the five defined classes - agent returns unclassified with the raw error summary and routes user to manual investigation
- Plan log is too large to fully process - agent processes the relevant error section and states this explicitly

**Duration**: Agent response time: target under 10 seconds [A]. For complex plan logs (>500 lines), target under 20 seconds [A].

---

### Stage 4: Workbench

**User Action**: User clicks "Open workbench" or "See proposed fix" from the inline panel. Navigates to the workbench view, which is scoped to this specific run and workspace.

**Agent Action**: Agent has already begun assembling the proposed fix in parallel with rendering the diagnosis. Workbench is populated with evidence panel, proposed fix diff, and speculative plan projection on arrival.

**Touchpoint**: Workbench - dedicated view within HCP Terraform, accessible from the run page. Run context is pre-loaded (user does not re-enter context).

**System Calls**:
- Session ID links workbench to the run page diagnosis (continuity across surfaces)
- AGX → Terraform MCP Server: read full workspace variable set, VCS repo (to generate the proposed change)
- AGX → GitHub (VCS connector): read the current file at the failing path [A: requires authenticated GitHub connector, AGX Priority 4]

**Data In** (workbench receives from agent):
- Evidence panel: full evidence set (log line, plan section, policy result, variable set, module reference, last successful run)
- Proposed fix diff: minimal change to fix the classified failure
- Speculative plan projection: predicted plan output if the fix is applied [A: speculative run triggered from the diff, not yet validated]
- Policy prediction: whether the fix is expected to pass the failing policy rule
- Confidence label and uncertainty notes (same as inline panel, expanded)
- Context chips: showing what the agent used - run, workspace, repo, variable set, policy set

**Data Out**: None in this stage. Read only.

**Validation**: Workbench must show the agent's context sources. User must be able to see which specific files, log lines, and policy rules the agent used. Unexplained fixes (where the reasoning is not traceable) are not acceptable.

**Failure Mode**:
- Workbench does not preserve run context from the run page - user has to start over (hard failure, must not happen)
- Agent cannot read the current file at the failing path (VCS not connected, or authenticated connector not available) - workbench shows what it can; fix diff is labeled "[A]: generated from log analysis only, not from current file state"
- Agent proposes a fix for a failure it misclassified - confidence label is the signal; evidence panel lets user verify or reject

**Duration**: Workbench loads pre-populated: under 2 seconds from click [A].

---

### Stage 5: Proposed Fix Review

**User Action**: User reviews the proposed fix diff line by line. Checks evidence panel to verify the fix is grounded. Reviews confidence label and uncertainty notes. Decides whether to proceed to validation or reject the fix.

**Agent Action**: Agent is in a waiting state. If the user asks a follow-up question in the workbench ("why did you change this line?"), the agent responds in context. [A: follow-up interaction is a post-MVP capability - MLE includes the static diff and evidence; interactive follow-up is post-MLE scope]

**Touchpoint**: Workbench - evidence panel (left or above), proposed fix diff (center), confidence/uncertainty notes (inline with diff).

**System Calls**: None new in this stage unless user asks follow-up question.

**Data In** (user reviews):
- Proposed fix diff: file path, changed lines, before/after
- Evidence panel: log line cited, plan section cited, policy rule cited, module reference
- Confidence label: high / medium / low with rationale
- Uncertainty notes: what the agent did not know
- Context chips: what sources the agent used

**Data Out**: None until user acts.

**Validation**: User is the validator at this stage. Design must make it possible to verify or refute each proposed change without leaving the workbench.

**Failure Mode**:
- User cannot trace the fix to the evidence (fix says "change variable X" but evidence panel does not show why variable X is the problem) - this is a design failure, not a system failure; the evidence panel must surface this explicitly
- Fix requires access to a file the agent could not read - diff is partial; labeled clearly
- User rejects the fix - workflow terminates here; audit records "user rejected proposed fix at workbench"; run page returns to failed state with no agent action taken

**Duration**: User-controlled. No time target.

---

### Stage 6: Validation (Speculative Run)

**User Action**: User clicks "Validate with speculative run" in the workbench. Reviews the speculative run result when it completes. If the speculative plan passes and policy clears, proceeds. If not, reviews the delta between the proposed fix and the remaining failure.

**Agent Action**: Agent applies the proposed diff to the current workspace config (in a temporary branch), triggers a speculative run via the HCP Terraform API, and waits for the result. When the result arrives, the agent interprets it: did the fix address the failure class? Did the policy pass?

**Touchpoint**: Workbench - speculative run result section. Result appears inline when complete; does not navigate away.

**System Calls**:
- Workbench → HCP Terraform API: trigger speculative run with the proposed diff applied [existing API capability]
- HCP Terraform → policy engine (Sentinel/OPA): run policy check on speculative plan
- Agent reads speculative plan result and policy result

**Data In**:
- Speculative plan output: pass/fail, changed resources, planned actions
- Policy result: pass/fail per policy, failing rule if any

**Data Out**:
- Speculative run result: "Speculative plan passed / failed - [N] resources changed"
- Policy result: "Policy passed / Policy [name] still failing: [rule]"
- Updated confidence label if result changes agent's assessment
- If validation passes: "Ready to create PR" state
- If validation fails: updated diagnosis with the delta failure

**Validation**: Speculative run result is the system validation. Design must surface the result clearly: pass means proceed; fail means review the delta and the agent re-diagnoses.

**Failure Mode**:
- Speculative run itself fails to trigger (API error) - surface clearly; do not silently skip this step
- Speculative plan passes but policy still fails - agent must surface this explicitly; do not route to PR creation if policy has not cleared
- Speculative plan passes and policy passes but the proposed diff is larger than the minimal fix - flag to user before PR creation

**Duration**: Speculative run: typically 30 seconds to 3 minutes depending on workspace size [A]. User waits in workbench with progress state visible.

---

### Stage 7: PR Creation

**User Action**: User clicks "Create PR" in the workbench. Reviews the pre-populated PR fields (title, description, evidence summary, validation status). Confirms or edits, then approves creation.

**Agent Action**: Agent generates the PR title, description, and body using the structured diagnosis, evidence citations, diff, and validation result. Creates the PR via the VCS connector (GitHub). Posts a PR comment with the diagnosis, evidence link back to the HCP Terraform workbench session, validation status, and suggested next action.

**Touchpoint**: PR creation dialog in workbench (modal or inline). GitHub PR and PR comment are the output surfaces.

**System Calls**:
- AGX → GitHub VCS connector: create branch with the proposed diff, open PR [requires GitHub VCS writeback, currently HCP-TF only - AGX Priority 1 for TFE]
- AGX → GitHub: post PR comment with structured diagnosis and workbench link

**Data In**:
- Proposed diff (from Stage 5)
- Speculative run result and policy result (from Stage 6)
- Evidence summary (from Stage 3)
- Workbench session ID (for the deep link in the PR comment)
- User identity (PR author)

**Data Out** (GitHub PR):
- PR title: "[Terraform] Fix [failure class] in [workspace name] - [date]" [A: agent-generated, user can edit]
- PR description: failure summary, evidence citations, fix rationale, validation result
- PR comment: shorter version - failure class, one evidence citation, validation status (speculative run passed, policy passed), link to HCP Terraform workbench session

**Validation**: PR must not be created if speculative run has not passed (system guard). PR must not be created if policy has not cleared (system guard). Both are hard gates.

**Failure Mode**:
- VCS connection not available (workspace not connected to VCS) - surface clearly; route user to "copy diff to clipboard" fallback path
- GitHub PR creation fails (permissions, branch conflict) - surface error; do not silently fail
- User edits PR description and removes evidence citations - no system guard needed; this is user choice

**Duration**: PR creation: under 5 seconds once user confirms [A].

---

### Stage 8: Review and Merge

**User Action**: Approver reviews the PR in GitHub. Reads the PR comment (diagnosis, evidence, validation status). Follows the workbench link if they want full evidence. Merges or requests changes.

**Agent Action**: None in this stage. The agent's work is complete. All outputs are in the PR.

**Touchpoint**: GitHub PR and PR comment (primary). HCP Terraform workbench session (available via deep link in PR comment).

**System Calls**: Standard GitHub PR review. If approver follows workbench link: HCP Terraform session lookup by session ID.

**Data In** (approver reads):
- PR comment: failure class, evidence citation, validation status, workbench link
- PR description: full evidence summary, fix rationale
- PR diff: the proposed change
- Workbench session (if followed): full evidence panel, speculative run result, agent context used

**Data Out**: Merge or change request.

**Validation**: Human approval is the validation at this stage.

**Failure Mode**:
- PR comment is a wall of text that mirrors the raw plan log - design failure in Stage 7; must be caught there
- Workbench link deep link is broken or session has expired - session should be retained for a minimum of 30 days [A]
- Approver merges without reading the evidence - this is acceptable; the audit trail records what was available

**Duration**: Approver-controlled. No time target.

---

### Stage 9: Audit

**User Action**: Terraform admin or security reviewer queries the agent session trace, either from the run page or from the admin audit view.

**Agent Action**: None in this stage. Audit data was recorded throughout the journey.

**Touchpoint**: Admin audit view - agent session trace scoped to run ID / workspace ID / session ID.

**System Calls**: Read from audit log (existing HCP Terraform run/workspace event store, extended with agent session fields).

**Data In**:
- Session ID
- Run ID, workspace ID
- User identity (who triggered the agent)
- Timestamp of each stage

**Data Out** (audit record shows)**:
- Intent: what the user asked the agent to do (trigger event, mode)
- Context used: which run, workspace, variable set, policy set, module version the agent read
- Tools used: which MCP tools the agent called
- Proposed diff: the exact change the agent proposed
- Validation result: speculative run and policy outcome
- Approval action: who approved PR creation, timestamp
- PR reference: link to the created PR

**Validation**: Audit record must be immutable. It must be visible to admins without requiring access to the agent session directly.

**Failure Mode**:
- Audit record is incomplete (missing context used or tools called) - this is a compliance failure; must be a hard requirement, not optional
- Audit record is not linked to the run/workspace history - admin cannot find it without knowing the session ID
- Agent session expires before audit record is queried - audit record must outlive the session

**Duration**: Admin-controlled. Query response: under 2 seconds [A].

## Data Flow

```
Run fails
    |
    v
[Stage 1: Trigger]
Run ID, plan log, policy result, workspace context, variable set, module source
    |
    v (user clicks "Ask Terraform")
[Stage 2: Request]
User identity + run context → AGX via manual webhook [AGX Priority 3 gate]
AGX → Terraform MCP: read run data
    |
    v (agent returns structured diagnosis)
[Stage 3: Diagnosis]
Failure class + evidence citations + confidence label → inline explanation panel
    |
    v (user opens workbench)
[Stage 4: Workbench]
Session ID preserved → workbench pre-populated
AGX reads current file state via VCS connector [AGX Priority 4 gate]
Evidence panel + diff + speculative plan projection → workbench surfaces
    |
    v (user reviews fix)
[Stage 5: Proposed Fix Review]
User validates evidence-to-fix traceability
    |
    v (user triggers speculative run)
[Stage 6: Validation]
Proposed diff → HCP TF API → speculative run
Speculative plan result + policy result → workbench
    |
    v (speculative pass + policy clear → user approves PR)
[Stage 7: PR Creation]
Diff + evidence + validation → AGX → GitHub VCS connector → PR + PR comment
PR comment embeds workbench session deep link
    |
    v (approver reviews)
[Stage 8: Review]
GitHub PR + PR comment + optional workbench deep link → human approval
    |
    v (throughout)
[Stage 9: Audit]
Intent + context + tools + diff + validation + approval → audit record
Linked to run ID + workspace ID + session ID
```
