## What This Doc Is

This strategy covers the near-term agentic direction for HCP Terraform: turning the platform's existing Terraform-native primitives (runs, plans, state, policy, registry, workspaces, audit) into a trusted intent-to-verification loop. It is the design and product strategy for a six-month build window beginning from the failed-run repair vertical slice. It enables downstream journey maps, PDRs, wireframe plans, and prototype validation.

Primary audience: HCP Terraform design, PM, engineering, and leadership workshop participants. Secondary audience: customer-facing teams aligned with Airbnb and similar accounts.

---

## The Use Case

A Terraform user encounters a failed run in HCP Terraform. Instead of digging through logs, they ask HCP Terraform what went wrong. The platform responds with a workspace-grounded explanation, cites specific evidence from the plan log and policy result, proposes the smallest safe fix, opens a workbench, generates a PR, runs speculative validation, and routes to approval. Every step is recorded in an audit trail. The same loop - grounded in Terraform-native context - is accessible from a run page, a GitHub PR comment, a Slack notification, or an external agent session using MCP.

The broader use case expands outward: module-backed intent to IaC, drift remediation, daily operational pulse, and workspace lifecycle automation. All share the same trust model and evidence chain.

## Scope

### In Scope

- Failed-run explanation with evidence citations (log lines, plan, policy result, workspace context, last successful run)
- Failed-run repair: proposed minimal fix for a defined set of failure classes
- Agent mode indicator on all entry surfaces: explain / plan / prepare PR / operate
- Workbench: evidence panel, proposed fix diff, speculative plan, policy result, approval controls
- PR creation and PR comment with diagnosis, evidence link, validation status, and next action
- Speculative run validation triggered from workbench
- Policy and approval gate on any write action
- Audit record: intent, context, tools used, diff, validation state, approval
- MCP setup and scoping UX: read/write scope, enabled tools, per-user token, approval boundaries visible
- Slack entry point: daily infra pulse with failed-run summary, workbench link, and action controls
- GitHub entry point: PR comment that routes to workbench and carries HCP Terraform evidence
- Terminal/CLI entry point: MCP handoff with visible scope
- Admin visibility: agent session trace, context/tools used, permission scope, evidence record
- Registry/module testing direction (not full delivery in first slice)
- Cross-surface continuity: start anywhere, verify in Terraform, evidence chain preserved

### Out of Scope

- Fully autonomous production apply without human approval
- Generic AI chatbot not grounded in Terraform context
- Full HCP Terraform UI/IA redesign as a prerequisite
- Blank-prompt multi-cloud provisioning as a first proof point
- Replacing GitHub code review
- Replacing policy-as-code (Sentinel / OPA)
- Arbitrary repo refactoring
- Hidden workspace variable mutation without approval
- Policy exception creation without explicit approval

### Scope Backlog (AGX-gated)

Design deliverables implied by the AGX 2026 priority roadmap that are not in the first slice. Each is blocked on a specific AGX platform capability, not a design decision. Design work can proceed on all four; production release is gated.

| Deliverable | What it is | AGX gate |
|---|---|---|
| Auto-invocation notification and approval flow | When an agent runs automatically on a run failure, policy gate, or native task event - the user needs to see it happened, understand what was triggered, and have a clear approval or dismiss path. ## Personas

Roles drawn from Airbnb feedback, documentation, and competitor analysis. No invented personas.

| Persona | Role | JTBD |
|---|---|---|
| Platform engineer | Owns the Terraform estate, workspace configuration, and policy compliance for the organization. | When a run fails in a critical workspace, I want to understand the root cause and get a reviewable fix without manual log triage so that I can unblock the team and preserve audit accountability. |
| App developer | Writes HCL for their service's infrastructure, depends on shared modules, and opens PRs. | When my Terraform PR triggers a speculative run failure, I want an explanation I can act on and a safe fix suggestion so that I can recover without escalating to the platform team. |
| Terraform admin / SRE | Monitors run health, drift, and policy compliance across many workspaces. | When the daily Slack pulse flags failures, drift, and policy violations, I want to see exactly what needs action and route each issue to the right person so that I can manage the estate without opening every workspace manually. |
| Terraform admin (security/compliance focus) | Audits agent actions, reviews permission boundaries, and approves changes at risk boundaries. | When an external agent uses HCP Terraform tools, I want to see exactly what it read, what it proposed, and what was approved so that I can satisfy audit requirements and catch out-of-bounds access. |

### Persona detail: Platform engineer

**Quote (assumed):** "I can tell a run failed. I can't always tell _why_ in under ten minutes, especially when a module or variable set is involved."

**Context:** Manages dozens to hundreds of workspaces. On-call for production failures. Needs triage speed, confidence in the proposed fix, and a reviewable artifact before any production change.

**Goals:** Reduce failed-run triage time. Recover without risk of introducing a second failure. Keep an audit trail the security team accepts.

**Pain points (Airbnb signal):** TFE doesn't explain failures; users dig through logs. No agent-assisted repair workflow. No evidence chain tying the fix back to the original failure.

**Behaviors:** Starts from run page or Slack notification. Wants evidence before touching production. Needs a PR they can review, not a direct apply.

**Design implication:** Explanation must cite specific evidence, not produce generic text. Workbench must show what the agent read and why it proposed the fix. PR must carry enough context to review without reopening the run page.

### Persona detail: App developer

**Quote (assumed):** "Speculative plan failed. The error references a module I didn't write. I have no idea which variable is wrong."

**Context:** Writes HCL within a monorepo, uses private registry modules, opens PRs targeting shared workspaces. May not have deep Terraform internals knowledge.

**Goals:** Understand failures without deep Terraform internals. Get a fix suggestion fast. Trust that the agent's PR won't break policy.

**Pain points:** Error messages reference module internals. GitHub PR comment shows "failed" with no diagnosis. Has to ask the platform team for every non-trivial failure.

**Design implication:** Inline PR comment must be self-contained and actionable. Evidence must surface the module source and workspace variable set in plain language. Mode must be visible so the developer knows the agent is proposing, not acting.

## Critical User Journeys

All CUJs trace to documented source material. Assumptions are labeled [A].

### CUJ-01 (Primary): Failed-run repair from run page

**Source:** `03-vertical-slice-failed-run-repair.md` - target user story; `04-workshop-questions-and-validation.md` - prototype validation plan

**Persona:** Platform engineer / app developer

**Trigger:** A run fails in HCP Terraform. User lands on the run page.

**Steps:**
1. User sees failed run with agent entry point visible - "Ask Terraform" or "Explain this failure"
2. Agent displays inline summary: failure classification, evidence citations (log line, plan section, policy result), workspace context (variable set, module source, last successful run)
3. User opens workbench
4. Agent shows proposed fix: changed files, diff, expected impact, speculative plan projection, policy pass/fail prediction
5. User reviews evidence panel: root cause, confidence label, uncertainty notes
6. User triggers speculative run validation from workbench
7. Speculative plan passes; policy gate clears
8. User approves PR creation
9. Agent opens GitHub PR with diagnosis, evidence link, validation status, and next action in PR comment
10. Approver reviews and merges PR
11. Audit trail records: intent, context used, tools read, diff, validation state, approval action

**Success criteria:**
- User can state the root cause of the failure without reading raw logs
- Proposed fix passes speculative run and policy gate
- PR comment is self-contained enough to review without opening HCP Terraform
- Audit record shows complete evidence chain

**Failure modes:**
- Agent produces generic text not tied to actual log/plan/policy evidence
- Workbench proposes a fix the user cannot verify (no evidence panel)
- PR is created but speculative run was not validated first
- Audit record is incomplete or not visible to admin

### CUJ-02: Failed-run diagnosis from GitHub PR comment

**Source:** `02-cross-surface-journeys.md` - section 3; `05-competitor-agentic-iac-analysis.md` - Scalr/Atlantis implication

**Persona:** App developer

**Trigger:** Speculative run fails on a GitHub PR. Developer sees a PR comment from HCP Terraform.

**Steps:**
1. PR comment shows: failure class, specific failed check, suggested action, link to HCP workbench
2. Developer clicks workbench link, lands in HCP Terraform with run context pre-loaded
3. Developer reviews evidence and proposed fix in workbench
4. Developer approves or rejects fix; audit records decision

**Success criteria:**
- PR comment is concise, specific, and actionable without opening HCP Terraform
- Workbench link preserves run context - user does not start over
- Fix suggestion maps to the exact failing check, not generic advice

**Failure modes:**
- PR comment is a wall of text that mirrors the raw plan log
- Workbench link drops the user at the workspace home with no context
- Suggested fix addresses a symptom, not the root cause

### CUJ-03: Daily Slack infra pulse - failed run triage

**Source:** `02-cross-surface-journeys.md` - section 4; `05-competitor-agentic-iac-analysis.md` - env zero/Firefly implication

**Persona:** Terraform admin / SRE

**Trigger:** Scheduled HCP Terraform analysis posts Slack pulse summary.

**Steps:**
1. Slack message lists: failed runs, drift events, policy failures, module risk flags, cost/security signals
2. Each item has: workspace name, failure class, severity, available actions (Explain / Open workbench / Create ticket / Dismiss)
3. Admin acts on high-priority items directly from Slack with policy-scoped actions
4. Actions post back to Slack with result; full record in HCP Terraform audit

**Success criteria:**
- Admin can triage estate health without opening HCP Terraform UI for each item
- Slack actions are constrained by policy and mirror HCP approval state
- Outcome is recorded in HCP Terraform, not only in Slack

**Failure modes:**
- Slack becomes a parallel control plane that diverges from HCP state
- Actions in Slack bypass approval gates
- Pulse is noisy - fires for every event, not summarized and prioritized

### CUJ-04: External agent (Copilot / Claude / terminal) using HCP Terraform context

**Source:** `01-six-month-vision.md` - section 3; `02-cross-surface-journeys.md` - section 2; `05-competitor-agentic-iac-analysis.md` - Geniez/domain substrate pattern

**Persona:** App developer (using an external coding agent), Terraform admin (auditing the session)

**Trigger:** Developer uses Claude, Copilot, or a terminal agent and invokes Terraform MCP tools.

**Steps:**
1. External agent requests HCP Terraform context via MCP
2. MCP setup panel shows: read/write scope, enabled tools, whose token applies, which actions require approval
3. Agent reads workspace, run, plan, policy data within declared scope
4. Agent proposes a change; write action requires explicit approval gate
5. Agent session trace recorded in HCP Terraform: what was read, what was proposed, what was approved
6. Admin can inspect the session trace from workspace/run history

**Success criteria:**
- User can see exactly what the external agent was allowed to read/write before the session starts
- Write actions require approval; approval boundary is not configurable away silently
- Admin can audit any external agent session from the HCP Terraform UI

**Failure modes:**
- MCP scope is invisible - user does not know what the agent can access
- External agent can apply changes without approval gate
- Session trace is not available or not tied to workspace/run history

### CUJ-05 (Supporting): Module-backed intent to IaC

**Source:** `04-workshop-questions-and-validation.md` - prioritization matrix, row 2; `03-vertical-slice-failed-run-repair.md` - section 2 (same rails)

**Persona:** Platform engineer / app developer

**Trigger:** User states an infrastructure intent (e.g., "provision an RDS database for the payments service in the dev workspace").

**Steps:**
1. Agent maps intent to available registry modules using HCP Terraform registry context
2. Agent produces module fit report: matching module, required variables, known risks, available test evidence
3. User selects module and confirms variable assignments
4. Agent generates HCL using the selected module
5. Speculative run validates the generated HCL
6. PR is created for review; module test evidence links to the PR

**Success criteria:**
- Generated HCL uses a registry module, not raw provider resources
- Module fit report surfaces variables, risks, and test evidence before the user commits
- PR is reviewable and speculative run passes before it is opened

**Failure modes:**
- Agent generates raw provider resources when a module exists
- Module fit report omits known risks or required variable context
- PR is created before speculative run validation

## Key Technical Context

### HCP Terraform existing assets

The following are available today and require no new primitives:

- **Run data**: plan log, apply log, status, timeline, error messages
- **Policy results**: Sentinel/OPA pass/fail with policy set context
- **Workspace context**: variable sets, linked VCS repo, module sources, last successful run, run history
- **State**: current state, state lineage
- **Registry**: private modules, provider versions, module test metadata
- **VCS integration**: branch, PR, commit SHA
- **Approvals**: approval state, approver identity
- **Audit log**: existing run/workspace event history
- **Terraform MCP Server**: registry, workspace, run management, tool filtering, per-user token passthrough, stdio and StreamableHTTP, rate limits, OTel metrics

**Gap:** The MCP Server has the raw capability. It is not yet wrapped in product UX: setup flow, scope visibility, tool narrowing in UI, approval boundaries, per-session trace, or hosted deployment with clear admin controls.

### Technical constraints from documentation

- [A] Agent classification of run failures must operate on existing log/plan/policy data - no new telemetry pipeline assumed for the first slice
- [A] PR creation uses existing VCS integration (VCS-connected workspaces); workspaces not connected to VCS need a separate path
- Speculative runs can be triggered today via the API; the workbench must expose this as a user-visible action
- Policy gates (Sentinel/OPA) are already part of the run lifecycle; the agent must interpret policy results, not bypass them
- [A] Audit trail is appended to existing run/workspace event log; no separate agent audit store assumed in the first slice
- MCP per-user token passthrough already exists; product UX for scope and tool enabling is the missing layer

### Failure classes for first slice (ordered)

1. Missing required variable
2. Invalid module input
3. Provider version conflict
4. Policy failure with clear remediation
5. Drift-caused plan surprise

### Surface entry points

- HCP Terraform run page (primary for CUJ-01)
- GitHub PR comment (primary for CUJ-02)
- Slack daily pulse (primary for CUJ-03)
- Terminal / external agent via MCP (primary for CUJ-04)
- HCP Terraform workbench (shared action layer for all CUJs)

## Open Questions

- Which workspaces in the first prototype have VCS connections? Does the slice work without VCS, or should the first demo assume VCS-connected workspaces? -- engineering
- What does the MCP per-user token UX look like today? Is there a setup flow, or is it raw config? -- engineering (Terraform MCP Server team)
- For the workbench: is it a modal, a side panel, or a dedicated page? What exists in the current HCP Terraform UI that could be the host? -- design / HCP UI team
- What is the approval model for agent-proposed changes? Is it a new approval type, or does it use existing workspace approval flows? -- PM / engineering
- Which Slack integration exists today: Webhook, Slack app with slash commands, or neither? What is the path to Slack actions that mirror HCP approval state? -- engineering
- For the audit trail: is the existing workspace/run event log sufficient, or does the agent session trace need its own data model? -- engineering / data
- How does the PR comment surface differ from what Scalr or Atlantis do today with HCP Terraform? Is there an existing PR comment integration to extend? -- engineering
- Airbnb follow-up needed: which exact failed-run classes were most painful, and which surfaces (terminal, GitHub, Slack, UI) are they operating agents from today? -- PM / customer success
- What confidence threshold is needed before surfacing a proposed fix? Is there a risk that low-confidence fixes erode trust faster than no-fix explanations? -- design / PM
- Module test evidence in the registry: what format does it need to be in to link from a PR or workbench? -- PM / registry team
