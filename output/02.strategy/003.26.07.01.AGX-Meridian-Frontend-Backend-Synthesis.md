## What This Doc Is

This document defines the separation of concerns between the AGX platform (backend agent infrastructure) and the HCP Terraform product surface (frontend UX). It exists to prevent scope confusion as design, product, and engineering work on agentic Terraform features: AGX owns the runtime and never surfaces directly to users; HCP Terraform's UI owns everything the user sees, touches, and acts on. Meridian's UX patterns inform HCP Terraform's interaction design exclusively - they do not define AGX behavior.

## Layer 1: AGX (Backend - Agent Runtime)

### What it is

AGX is the managed agent platform owned by Marc Cosentino. It is infrastructure. Users do not interact with it directly. They receive its outputs through HCP Terraform's UI surfaces. AGX is defined in the AGX 2026 Priorities memo as:

> A managed agent is an AI agent designed, hosted, operated, and maintained by HashiCorp/IBM as part of the Terraform product experience. It runs within HashiCorp/IBM-controlled infrastructure and is triggered directly from Terraform workflows. The agent's implementation is not exposed to customers.

### What AGX owns

| Capability | Description |
|---|---|
| Agent orchestration | Multi-step workflow: planning, tool use, state transitions, retries, approvals, result reporting |
| Trigger framework | Webhooks (manual, run/native task, notification), scheduled execution |
| Connector and auth layer | Terraform MCP, Terraform CLI, tfctl, GitHub, customer-supplied MCPs |
| Skills | Domain-specific actions, tools, prompts, guardrails in composable units |
| Multi-agent coordination | Primary agent + sub-agents; context and ownership preserved across handoffs |
| Model selection | Choose model per task: Claude, GPT, Nova, etc. based on latency, cost, capability |
| Bring-your-own provider | Customer-supplied model provider keys (Bedrock, Azure Foundry, Vertex, Anthropic, OpenAI) |
| Context, memory, and RAG | Workspace metadata, run history, policy context, documentation at execution time |
| Vector and contextual storage | Embeddings with freshness, retention, and tenant isolation |
| Execution environments | Reproducible, auditable environments per agent class |
| Sandboxing and credential isolation | Scoped credentials, least-privilege, blast-radius control |
| AGX Bridge | Identity delegation across agent workflows |
| Execution scalability | Thousands of concurrent agent executions; horizontal scaling; tenant isolation |
| Billing and metering | Token usage, model calls, tool execution metering for chargeback and future pricing |
| Agentic SDK | Developer framework for building agents, sub-agents, triggers, connectors, policies |

### Current state (as of June 2026)

- Integrated with HCP-TF only. TFE not yet supported.
- Single tenancy only. Multi-tenancy in progress.
- MCP connector uses static credentials. Authenticated MCP is desired state, not current.
- GitHub VCS writeback exists for HCP-TF. TFE writeback not yet supported.
- Trigger support is generic webhooks only. Manual UI triggers, run/native task triggers, notification triggers, and scheduled triggers are all desired state.
- Model access constrained to AWS Bedrock-enabled models. BYOP not yet implemented.

### What AGX does not own

- How outputs are displayed to users
- How users initiate requests (button labels, modal design, page placement)
- How approval gates look and behave in the UI
- How audit evidence is surfaced
- How mode indicators are shown
- Any HCP Terraform page layout or information architecture

## Layer 2: HCP Terraform Frontend (User-Facing Surface)

### What it is

The HCP Terraform UI is the surface where users initiate, monitor, review, approve, and audit agent actions. It calls AGX via the trigger boundary and receives AGX outputs to render. The user's entire experience of agentic Terraform is defined here. Design work happens here exclusively.

### What the HCP Terraform frontend owns

| Surface | What it owns |
|---|---|
| Run page entry point | Inline "Ask Terraform" / "Explain this failure" button, trigger to AGX |
| Inline explanation panel | Renders AGX output: failure class, evidence citations, workspace context |
| Mode indicator | Displays current agent mode: explain / plan / prepare PR / operate |
| Context chips | Shows what the agent is using: run, workspace, repo, policy, module, variable set |
| Workbench | Evidence panel, proposed fix diff, speculative plan result, policy result, approval controls |
| Approval gate | Surfaces write-action boundary; shows what the agent is requesting permission to do |
| PR comment | Renders AGX diagnosis, evidence link, validation status, and next action in GitHub |
| Slack pulse | Presents AGX-generated daily summary; routes to workbench |
| MCP scope panel | Shows read/write scope, enabled tools, token identity, approval boundaries |
| Admin audit view | Agent session trace: what was read, proposed, approved; tied to run/workspace history |

### What the HCP Terraform frontend does not own

- How the agent reasons about a failure
- Which model is called
- How context is retrieved or ranked
- How multi-agent handoffs work
- Token costs or model limits
- Agent execution environments or sandboxing

## Meridian UX Patterns: Frontend Only

### What Meridian is

Meridian is a separate product scoped to Terraform for Z (IBM Z / LinuxONE). As of June 2026 it is narrowing to a Terraform for Z add-on, not a standalone launch. The Meridian product itself is not the HCP Terraform frontend.

However, the Meridian PRD defines a set of UX patterns that are directly applicable to HCP Terraform's agentic surfaces. These patterns are the relevant output for HCP Terraform design work.

### Meridian UX patterns applicable to HCP Terraform

These are borrowed as interaction design patterns only. They do not represent a product dependency on Meridian.

**Chat and natural language entry (MER-16)**
- Model selection visible to user
- Agent selection when agents are visible
- Session history retained and browsable
- Connection status displayed (backend connected / disconnected)
- Clear chat control
- Confidence score on every response
- Response time target: under 10 seconds

**Workflow browser (MER-27)**
- Browse and search available workflows/agents
- Corresponds to the HCP Terraform workbench: user can see which agent use cases are available and select one
- Post-GA: download, customize, and upload workflows (maps to customer-supplied Skills in AGX)

**Audit trail (MER-23)**
- Every prompt logged
- Every action logged
- User identity recorded
- Logs are searchable and immutable
- Maps to the HCP Terraform admin audit view for agent sessions

**AI transparency (MER-92, Post-GA)**
- Sources displayed with every response
- Reasoning summary provided
- Confidence score visible
- User feedback mechanism
- Maps to the evidence panel in the HCP Terraform workbench

**Observability dashboard (MER-93)**
- Query volumes, latency, error rates, agent activity visible to admins
- Data exportable
- Maps to the HCP Terraform admin visibility requirements in the strategy doc

**Agent governance controls (MER-94, Post-GA)**
- Admin can approve/configure which agents are available
- Agent permissions are configurable
- Risky actions require approval
- Maps to the MCP scope panel and approval gate in the HCP Terraform strategy

**Identity and access (Epic 8)**
- OIDC / SAML authentication
- RBAC across the application
- Maps to the enterprise permission model already in HCP Terraform

### What Meridian patterns do not apply to

Meridian is built for IBM Z mainframe operations. Its domain-specific agents (Operations Agent, Developer Agent, Security Agent, Contextual Operational Assistant) are Z-specific and do not transfer. Only the interaction design patterns above are relevant to HCP Terraform.

## The Trigger / Response Boundary

This is the seam between the two layers. It must be explicitly designed.

### Trigger: UI → AGX

AGX's 2026 trigger roadmap defines how the UI calls AGX:

| Trigger type | Use case in HCP Terraform UI |
|---|---|
| Manual webhook (desired) | "Ask Terraform" button on run page; "Open workbench" action |
| Run/native task webhook (desired) | Automatic agent invocation when a run fails a policy or plan |
| Notification webhook (desired) | Slack pulse triggered by workspace events (failed run, drift, policy failure) |
| Scheduled trigger (desired) | Daily infra pulse generation |

All four are desired state, not current. The manual webhook is the most critical for the failed-run repair slice.

### Response: AGX → UI

AGX returns agent output. The HCP Terraform UI is responsible for rendering it. The output contract should include at minimum:

| Field | What it is | Where it surfaces in UI |
|---|---|---|
| Failure classification | Category of failure (missing variable, policy violation, etc.) | Inline explanation panel |
| Evidence citations | Specific log lines, plan sections, policy results cited | Evidence panel in workbench |
| Proposed fix | Changed files, diff, expected impact | Proposed fix panel in workbench |
| Confidence label | How confident the agent is in the diagnosis and fix | Inline and workbench |
| Uncertainty notes | What the agent does not know or cannot verify | Evidence panel |
| Validation result | Result of speculative plan, policy check | Validation section in workbench |
| Agent mode | What mode was active: explain, plan, prepare PR, operate | Mode indicator |
| Tool scope used | What the agent read/wrote during the session | MCP scope panel, audit view |
| Session ID | Unique identifier linking all surfaces to one agent session | Audit view, PR comment link |

## Scope rules for design and engineering

| If you are working on... | It belongs to... | Owner |
|---|---|---|
| How the agent reasons, retrieves, or executes | AGX backend | Marc Cosentino / AGX team |
| How the user sees, triggers, reviews, or approves | HCP Terraform frontend | Design / HCP TF product team |
| Meridian Z-specific agents and workflows | Meridian product | Paul, Tiffany, Saurabh, Chris B |
| Meridian interaction patterns applied to HCP TF UI | HCP Terraform frontend | Design (Gabrielle / Ange) |
| The trigger and response contract between layers | Shared - must be co-designed | AGX + HCP TF product + design |

## Workshop implications

- Design's six-month scope (workbench, run page entry, PR comment, Slack pulse, mode indicator, audit view, MCP scope panel) is entirely frontend. It does not depend on AGX internal decisions.
- The BYOP vs. managed provider open decision in AGX has no UI consequence until it is resolved, except for the MCP scope panel which must show which provider is active.
- The trigger boundary is the critical path dependency: the manual webhook trigger (desired state in AGX) must land before the "Ask Terraform" button in the UI can call a real agent. Design can proceed against a mock before this is ready.
- Multi-tenancy being in progress in AGX is a constraint on when multi-tenant HCP-TF features can go to production, but it does not block design or prototype work.
- The Meridian PRD's chat interface requirements (confidence score, session history, connection status, model selection) should be the starting point for the HCP Terraform workbench interaction spec - they are already written and validated in a parallel product context.
