## What This Doc Is

This document answers one question: given the five AGX 2026 priorities (in order), which HCP Terraform UI surfaces and Meridian interaction patterns can design build now versus which are blocked, and on what?

It is a sequenced execution readiness map for the six-month design window. It does not restate the layer separation (see `003`). It does not restate the strategy CUJs (see `002`). It maps AGX priority delivery to design buildability.

---

## Reading the map

Each AGX priority is presented in the order given in the 2026 memo. For each:
- **What AGX delivers** - the platform capability the priority produces
- **UI surfaces it unlocks** - what the HCP Terraform frontend can now build or go to production with
- **Meridian patterns it enables** - which Meridian interaction design patterns (from the PRD) become applicable
- **Design action** - what design can do now, and what to defer

**Current state baseline:**
- Triggers: generic webhooks only (manual, run/task, notification, scheduled: all desired state)
- MCP connector: static credentials only
- HCP-TF integration: exists; TFE: not yet
- Multi-tenancy: in progress
- Model access: AWS Bedrock only; BYOP: desired state

Design can proceed against any surface where the trigger and response contract can be mocked. Design cannot ship to production on any surface that requires a trigger type not yet supported or authenticated MCP.

## Priority 1: TFE Support

**What AGX delivers**
Full integration with customer self-hosted Terraform Enterprise environments. Agents can be invoked from TFE workspaces, write back to TFE GitHub VCS connections, and operate against TFE run history and state.

**UI surfaces it unlocks**

| Surface | Current state | After TFE support |
|---|---|---|
| Run page entry point ("Ask Terraform") | HCP-TF only | Available on TFE run pages |
| Workbench | HCP-TF workspaces only | Available for TFE workspaces |
| PR comment | HCP-TF VCS connections only | Available for TFE VCS connections |
| Admin audit view | HCP-TF only | Available for TFE organizations |

**Meridian patterns it enables**
No new Meridian patterns are unlocked by TFE support specifically. TFE support extends existing HCP-TF surfaces to a second deployment context. All Meridian patterns that apply to HCP-TF (MER-16 chat entry, MER-23 audit trail, MER-27 workflow browser, MER-92 AI transparency, MER-93 observability, MER-94 governance) apply equally to TFE once the integration exists.

**Design action**
- Design HCP-TF surfaces first. TFE surfaces are the same interaction design in a different deployment context; they do not require separate wireframes unless TFE has structural UI differences.
- Flag any TFE-specific admin controls (org-level agent enablement, on-prem policy controls) that require a separate admin settings section in TFE.
- Do not block HCP-TF design work waiting for TFE support.

## Priority 2: Multi-Tenancy

**What AGX delivers**
Single tenancy for TFE, multi-tenancy for HCP-TF. Agents operate in isolated execution environments per organization. Credentials, context, and outputs are isolated across tenants.

**UI surfaces it unlocks**

| Surface | Current state | After multi-tenancy |
|---|---|---|
| MCP scope panel | Shows single-tenant token scope | Must show per-organization scope boundary clearly |
| Admin audit view | Can go to production for HCP-TF | Audit records are now tenant-isolated; safe for production in multi-tenant HCP-TF |
| Agent governance controls | Limited pre-multi-tenancy | Full org-level agent enablement and permission controls can go to production |

**Meridian patterns it enables**

**MER-94: Agent governance controls (Post-GA)**
- Admin-configurable agent availability per organization
- Organization-scoped agent permissions
- Approval gates on write-class actions
- This pattern requires tenant isolation to be safe in production. Multi-tenancy in AGX is the prerequisite.

**Identity and access (Epic 8)**
- RBAC across the agentic surface is meaningful only when tenants are isolated.
- Organization-level role assignment for agent features becomes a real product control after multi-tenancy lands.

**Design action**
- Design the MCP scope panel and admin audit view now. These interact designs do not change based on tenancy model.
- Defer production release of org-level agent governance controls until multi-tenancy is confirmed complete in HCP-TF.
- Design the agent enablement admin settings page now; mark it as staged release dependent on multi-tenancy.
- The governance controls UX (which agents are available, which require approval, which roles can invoke them) is a design-complete deliverable that waits for an AGX backend gate, not a design gate.

## Priority 3: Additional Agent Triggers

**What AGX delivers**
This is the highest-impact AGX priority for design. Four new trigger types beyond generic webhooks:

| Trigger type | What it enables in UI |
|---|---|
| Manual webhook (UI button) | The "Ask Terraform" / "Explain this failure" button on the run page calls a real agent |
| Run/Native Task webhook | Automatic agent invocation on run failure, policy gate, or native task event |
| Notification webhook | Slack pulse triggered by workspace events (failed run, drift, policy violation) |
| Scheduled trigger | Daily infra pulse at a set cadence |

**UI surfaces it unlocks**

| Surface | Blocked on | Unlocked when |
|---|---|---|
| Run page entry point | Manual webhook | Manual webhook lands |
| Workbench (full session) | Manual webhook | Manual webhook lands |
| Auto-invocation on run failure | Run/native task webhook | Run/native task webhook lands |
| Slack pulse (event-driven) | Notification webhook | Notification webhook lands |
| Daily infra pulse | Scheduled trigger | Scheduled trigger lands |
| PR comment | Run/native task or manual webhook | Either webhook lands |

**Meridian patterns it enables**

**MER-16: Chat and natural language entry**
- The manual webhook is the mechanism that fires the agent from a user's explicit request. Until the manual webhook exists, chat-style entry into the agent has no real backend. Design can build the UI; it cannot call a live agent.
- Model selection and session history (MER-16 sub-requirements) are frontend concerns that do not depend on the trigger type.

**MER-27: Workflow browser**
- "Browse available agent workflows" requires there to be invokable agent workflows. Manual webhook landing makes one workflow real (failed-run repair). Subsequent trigger types add more.

**Design action**
- Design all trigger-dependent surfaces now against mock responses. The trigger/response API contract (defined in `003`) is what the mock must implement.
- For the manual webhook: agree the API contract with AGX before the button is built. The button's existence and label are design decisions; what it calls is an AGX decision; what comes back is a shared contract.
- Prioritize the manual webhook interaction in workshop discussion with Omar and Daniel. This is the single point of integration between design's six-month scope and AGX's 2026 work.
- The run/native task webhook unlocks auto-invocation - design the "agent ran automatically on your behalf" notification and approval flow now; it will need to be ready when AGX ships this trigger type.

## Priority 4: Connectors and Integrations (Authenticated Skills)

**What AGX delivers**
Two main outcomes for design:
1. **Authenticated Terraform MCP connector** - seamless auth for the MCP server, replacing static credentials. The agent can now act against Terraform with a scoped, authenticated identity rather than a static token.
2. **Customer-supplied Skills** - customers can add their own skills to the agent. This introduces a new class of admin UI: skill management.

Secondary outcomes: Terraform CLI + tfctl authentication, TFE VCS writeback, Terraform Skills (HCL-writing, module-building, policy-writing).

**UI surfaces it unlocks**

| Surface | Current state | After authenticated connectors |
|---|---|---|
| MCP scope panel | Shows static credential scope | Shows per-user, scoped token identity; meaningful trust signal to user |
| Approval gate | Works but trust context is weak | Can show authenticated identity doing the write action; stronger trust signal |
| Admin audit view | Agent session logged | Logs now carry authenticated identity, not a static token reference |
| Skills management (new) | Does not exist | Admin surface to view, enable, and configure available skills including customer-supplied ones |

**Meridian patterns it enables**

**MER-92: AI transparency (Post-GA)**
- Authenticated MCP means the "sources displayed with every response" pattern (MER-92) can reference real, authenticated tool calls rather than opaque static-credential executions. The transparency claim is stronger.
- Confidence scores and reasoning summaries (MER-92) are independent of auth and can be designed now.

**MER-94: Agent governance controls - extended**
- Customer-supplied Skills require an admin governance surface: which skills are available, who approved them, what their scope is. This is the Skills Management page.
- This is a new design deliverable not present in the current strategy doc `002` scope. It should be added to the scope backlog.

**Design action**
- Design the MCP scope panel showing authenticated identity as the production-ready state. The current static-credential state can be reflected in the design as a transitional state with a visual indicator (e.g., "using shared credential" vs. "using your credential").
- Add Skills Management to the scope backlog as a post-P4 deliverable: admin page to view active skills, enable/disable customer-supplied skills, see skill scope and approval status.
- The approval gate interaction design is not blocked on authenticated MCP. Design it now; the identity field in the approval gate will populate correctly once auth lands.

## Priority 5: Execution Scalability

**What AGX delivers**
Reliable handling of potentially thousands of concurrent agent executions. Horizontal scaling, durable job queuing, prioritization, pause/resume/cancel, and tenant isolation under load.

**UI surfaces it unlocks**

| Surface | Current state | After scalability work |
|---|---|---|
| Run page entry point | Works at low volume | Works at enterprise scale; no change to UI behavior |
| Workbench | Works for one session | Concurrent sessions across many workspaces become reliable |
| Admin observability dashboard | Useful at low volume | Meaningful at enterprise scale; volume metrics become real signals |

**Meridian patterns it enables**

**MER-93: Observability dashboard**
- Query volumes, latency, error rates, and agent activity metrics are only meaningful when agents run at scale. Scalability landing is what makes the observability dashboard a real operator tool rather than a demonstration artifact.
- The dashboard can be designed now. The data will populate fully only after scalability work is complete.

**Design action**
- Scalability has no direct UI design dependency. Design does not need to wait on this priority for any interaction surface.
- Include agent queue status and execution state in the admin observability dashboard design (job queued, running, paused, cancelled) as this reflects the durable queuing model AGX is building toward.
- Response-time targets in the Meridian PRD (under 10 seconds for MER-16) become a real contract once agents run at scale. Design should set user-visible loading/progress states that work at p50 and p99 latency.

## Sequenced readiness view

This is the design buildability timeline based on AGX priority delivery order.

```
NOW (design can proceed)
├── Run page entry point UI (button, label, mode indicator) - mock only
├── Inline explanation panel - mock response
├── Workbench layout (evidence, diff, plan, approval controls) - mock
├── PR comment structure and routing - mock
├── MCP scope panel (transitional: static credential state)
├── Admin audit view structure and interaction
├── Meridian patterns: MER-16 layout, MER-23 audit, MER-92 transparency UI
│
├── AFTER Priority 1 (TFE support):
│   └── Extend all HCP-TF surfaces to TFE context; no new wireframes required
│
├── AFTER Priority 2 (multi-tenancy):
│   ├── Agent governance admin controls go to production (MER-94)
│   └── Org-scoped agent enablement settings page ships
│
├── AFTER Priority 3 (triggers): ← CRITICAL PATH for first real integration
│   ├── Run page button calls real agent (manual webhook)
│   ├── Auto-invocation on run failure wires up (run/native task webhook)
│   ├── Slack pulse becomes event-driven (notification webhook)
│   └── Daily infra pulse becomes scheduled (scheduled trigger)
│
├── AFTER Priority 4 (connectors/skills):
│   ├── MCP scope panel shows authenticated identity (not static credential)
│   ├── Approval gate shows authenticated identity
│   ├── Audit view carries real identity references
│   └── Skills Management admin page ships
│
└── AFTER Priority 5 (scalability):
    └── Observability dashboard data becomes real at enterprise volume
```
