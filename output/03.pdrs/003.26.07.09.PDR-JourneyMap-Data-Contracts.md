## Problem Statement

When a run fails, the platform engineer uses multiple systems (HCP Terraform, AGX, GitHub, Slack) across a single agent session. Each system transition requires a consistent data contract: what goes in, what comes out, and what must be preserved across the boundary. Without a specified contract, each system boundary becomes an integration risk - session context drops, evidence citations are truncated, speculative run results do not match the proposed diff, or the audit record is incomplete. This PDR defines the per-stage data contracts, state machine, service map, and timing budget for PDR-002's scenario as the single source of truth for implementation.

---

## Journey Overview

Nine stages. Four system boundaries that require explicit contract specification: UI-to-AGX trigger, AGX-to-UI diagnosis, UI-to-HCP-TF speculative run, and AGX-to-GitHub PR creation. The session ID is the primary cross-boundary identifier. The audit record must be populated continuously from Stage 2 onward.

```
Stage 1: Trigger         - Run data at rest; no AGX call
Stage 2: Request         - UI → AGX trigger; AGX → MCP read
Stage 3: Diagnosis       - AGX → UI diagnosis payload
Stage 4: Workbench       - Session context preserved; AGX → VCS read [gated]
Stage 5: Fix Review      - User decision; optional diff edit
Stage 6: Validation      - UI → HCP TF API speculative run; HCP TF → policy engine
Stage 7: PR Creation     - AGX → GitHub VCS connector (branch + PR + comment)
Stage 8: Review/Merge    - Human action; no agent calls
Stage 9: Audit           - HCP TF audit log read
```

## Full Journey Map (Data Contracts)

### Stage 1: Trigger

**Inputs:**
| Field | Source | Required | Notes |
|---|---|---|---|
| `run_id` | HCP Terraform run record | Yes | Used in all downstream calls |
| `workspace_id` | HCP Terraform workspace | Yes | Scopes all MCP reads |
| `status` | HCP Terraform run record | Yes | Must be `errored` to surface "Ask Terraform" |
| `plan_log_url` | HCP Terraform run record | Yes | Resolved in Stage 3 |
| `policy_result` | HCP Terraform run record | Yes | Pass/fail and policy set context |
| `vcs_commit_sha` | HCP Terraform workspace VCS | Conditional | Required if VCS-connected workspace |
| `variable_set_ids[]` | HCP Terraform workspace | Yes | Used in Stage 3 variable absence check |

**Outputs:** None. Stage 1 is observation only.

**Services:** HCP Terraform data store (read, existing capability)

**Validations:**
- `status == errored`: "Ask Terraform" button is visible only when this is true. Button must not surface on passing runs.

---

### Stage 2: Request

**Inputs:**
| Field | Source | Required | Notes |
|---|---|---|---|
| `session_id` | UI-generated (UUID) | Yes | Primary cross-boundary identifier |
| `run_id` | Stage 1 | Yes | |
| `workspace_id` | Stage 1 | Yes | |
| `user_identity` | MCP token passthrough | Yes | Establishes read/write scope |
| `mode` | User selection | Yes | Default: `explain`. Range: `explain \| plan \| prepare_pr` |
| `trust_level` | Org/workspace setting | Yes | L1 or L2 at MVP |

**Outputs (trigger payload - UI → AGX):**
```json
{
  "session_id": "string (UUID)",
  "run_id": "string",
  "workspace_id": "string",
  "user_identity": "string (MCP token reference)",
  "mode": "explain | plan | prepare_pr",
  "trust_level": "L1 | L2"
}
```

**Outputs (trigger acknowledgment - AGX → UI):**
```json
{
  "session_id": "string (echoed)",
  "status": "processing",
  "estimated_response_seconds": "integer"
}
```

**Services:**
- UI → AGX: manual webhook [AGX Priority 3, desired state]
- AGX → Terraform MCP Server: begins context assembly reads (results returned in Stage 3)

**Validations:**
- `session_id` must be unique per session invocation. Reuse is not permitted.
- `mode` must not exceed the `trust_level`. L1 orgs may not send `mode: prepare_pr`.
- MCP token must have read scope on the workspace before trigger fires. Scope failure surfaces in the MCP scope panel before the button activates, not after.

**UI state after Stage 2:**
- Mode indicator: "Agent working - [mode] mode"
- Spinner: visible, with `estimated_response_seconds` countdown
- No write action has occurred

### Stage 3: Diagnosis

**Inputs (AGX reads via Terraform MCP):**
| Field | MCP tool | What AGX reads |
|---|---|---|
| Plan log error section | `get_run` | Error lines only, not the full log |
| Policy results | `get_run` | Failing policy name, rule text, cited resource |
| Variable set | `get_variable_set` | Variable names, presence/absence, required flag |
| Module metadata | `get_module_metadata` | Input schema, required fields, default values |
| Run history | `get_workspace_run_history` | Last successful run ID and timestamp |

**Outputs (diagnosis payload - AGX → UI):**
```json
{
  "session_id": "string",
  "failure_classification": "missing_variable | invalid_module_input | provider_version_conflict | policy_failure | drift_surprise | unclassified",
  "evidence_citations": [
    {
      "type": "log_line | variable_absence | policy_rule | module_reference | drift_delta",
      "source": "plan_log | variable_set | policy_result | registry | run_history",
      "reference": "string (artifact ID or path)",
      "excerpt": "string (exact text cited, not paraphrased)"
    }
  ],
  "confidence_label": "high | medium | low",
  "confidence_rationale": "string",
  "uncertainty_notes": ["string"],
  "proposed_fix_diff": null,
  "agent_mode": "explain | plan | prepare_pr",
  "tool_scope_used": [
    {
      "tool_name": "string",
      "action": "read | write",
      "artifact_reference": "string"
    }
  ],
  "workspace_context": {
    "variable_set_id": "string",
    "module_sources": ["string"],
    "vcs_repo": "string | null",
    "last_successful_run_id": "string | null"
  }
}
```

**Services:** AGX, Terraform MCP Server (read-only)

**Validations:**
- Every `evidence_citations` entry must reference a real artifact that exists in the run data. Generic text not tied to a specific artifact must not appear in citations.
- `confidence_label: low` requires at least one entry in `uncertainty_notes`.
- `failure_classification: unclassified` must include the raw error excerpt in evidence citations.
- `proposed_fix_diff` must be `null` in `explain` mode.

### Stage 4: Workbench

**Inputs:**
| Field | Source | Required | Notes |
|---|---|---|---|
| Diagnosis payload | Stage 3 | Yes | Evidence citations, confidence label, context |
| `session_id` | Stage 2 | Yes | Must be preserved; used to pre-populate workbench |
| Full variable set | Terraform MCP `get_variable_set` | Yes | Needed to generate diff |
| Current file at failing path | GitHub VCS connector `read_file` | Conditional | Required for accurate diff; [A: AGX Priority 4 gate] |

**Outputs (workbench pre-populated payload):**
```json
{
  "session_id": "string",
  "evidence_panel": {
    "citations": "[same structure as evidence_citations from Stage 3]",
    "context_chips": [
      { "type": "run | workspace | variable_set | module | policy_set | repo", "label": "string", "reference": "string" }
    ]
  },
  "proposed_fix_diff": {
    "files": [
      {
        "file_path": "string",
        "before": "string",
        "after": "string",
        "diff_source": "vcs_connector | variable_set_analysis",
        "diff_source_note": "string | null"
      }
    ]
  },
  "speculative_plan_projection": {
    "expected_resource_changes": "integer",
    "projected_resources": ["string"],
    "is_projection": true
  },
  "policy_prediction": {
    "expected_result": "pass | fail | unknown",
    "rationale": "string"
  },
  "confidence_label": "high | medium | low",
  "uncertainty_notes": ["string"]
}
```

If VCS connector is not available: `diff_source` is `variable_set_analysis` and `diff_source_note` is set to a user-visible explanation. The diff is still shown; the label makes clear it is estimated.

**Services:** AGX, Terraform MCP Server (read), GitHub VCS connector (read, conditional)

**Validations:**
- `session_id` must match the session started in Stage 2. Workbench must not load without a valid session.
- `proposed_fix_diff` must not be empty if `mode` is `plan` or `prepare_pr`.

---

### Stage 5: Fix Review

**Inputs:** Workbench state from Stage 4. User reads and optionally edits the diff.

**Outputs (if user edits diff):**
```json
{
  "session_id": "string",
  "user_modified_diff": true,
  "original_diff": "{ same structure as proposed_fix_diff }",
  "modified_diff": "{ same structure as proposed_fix_diff, with user changes }"
}
```

**Outputs (if user rejects):**
```json
{
  "session_id": "string",
  "terminal_stage": 5,
  "terminal_action": "user_rejected_fix"
}
```

Session terminates. Audit record is closed at this stage. No write action has occurred.

**Services:** None (read-only; user decision stage)

**Validations:** None from the system. User is the validator.

---

### Stage 6: Validation

**Inputs:**
| Field | Source | Required | Notes |
|---|---|---|---|
| `workspace_id` | Stage 1 | Yes | |
| Modified or proposed diff | Stage 4 or Stage 5 | Yes | The diff that will be validated |
| `session_id` | Stage 2 | Yes | Links speculative run to session |

**Outputs (speculative run trigger - Workbench → HCP Terraform API):**
```json
{
  "workspace_id": "string",
  "variables": "{ key: value pairs from diff }",
  "is_speculative": true,
  "message": "Agent-proposed fix validation - session [session_id]"
}
```

**Outputs (speculative run result - HCP Terraform → Workbench):**
```json
{
  "speculative_run_id": "string",
  "plan_status": "planned | errored",
  "resources_changed": "integer",
  "planned_resource_actions": ["string"],
  "policy_status": "passed | failed | not_run",
  "failing_policy_name": "string | null",
  "failing_policy_rule": "string | null"
}
```

**Outputs (agent interpretation - AGX → UI):**
```json
{
  "session_id": "string",
  "speculative_run_id": "string",
  "speculative_plan_result": "passed | failed",
  "resources_changed": "integer",
  "policy_result": "passed | failed",
  "failing_policy_name": "string | null",
  "updated_confidence_label": "high | medium | low | null",
  "ready_for_pr": "boolean"
}
```

**Services:** HCP Terraform API (speculative run, existing capability), Policy engine Sentinel/OPA (invoked automatically by speculative run)

**Validations:**
- `ready_for_pr` must be `true` only when: `speculative_plan_result == passed` AND `policy_result == passed`.
- `ready_for_pr: false` must disable the "Create PR" button in the UI. This is a hard guard.
- If speculative run itself errors (API failure), surface error state. Do not set `ready_for_pr: true` on an unvalidated diff.

---

### Stage 7: PR Creation

**Inputs:**
| Field | Source | Required | Notes |
|---|---|---|---|
| `session_id` | Stage 2 | Yes | Embedded in PR comment deep link |
| Diff (modified or proposed) | Stage 5 | Yes | |
| Speculative run result | Stage 6 | Yes | `ready_for_pr` must be `true` |
| Evidence summary | Stage 3 | Yes | Included in PR description |
| `user_identity` | Stage 2 | Yes | PR author attribution |
| VCS repo | Stage 1 / workspace context | Yes | |

**Outputs (branch and PR creation - AGX → GitHub VCS connector):**
```json
{
  "repo": "string",
  "base_branch": "string",
  "new_branch": "terraform-agent/[slug]-[run_id]",
  "files_changed": [
    { "file_path": "string", "content": "string (full file content with diff applied)" }
  ],
  "pr_title": "string",
  "pr_description": "string (markdown; includes evidence summary, fix rationale, validation result)",
  "pr_labels": ["terraform-agent-fix"],
  "session_id": "string"
}
```

**Outputs (PR comment - AGX → GitHub):**
```json
{
  "pr_number": "integer",
  "comment_body": "string (markdown; shorter than PR description; includes deep link to workbench session)"
}
```

Workbench deep link format: `https://app.terraform.io/workbench/{session_id}`

**Services:** AGX, GitHub VCS connector (write; branch creation, PR creation, comment posting)

**Validations (hard guards):**
- PR creation must be blocked if `ready_for_pr` is not `true`.
- PR creation must be blocked if speculative run was not completed in Stage 6.
- PR comment must include the `session_id` deep link. A PR created without the deep link is a compliance failure.
- PR description must include at minimum: failure classification, at least one evidence citation, validation result (speculative run and policy), and session deep link.

**Failure modes and fallbacks:**
- VCS not connected: PR creation path is unavailable. Surface "Copy diff" action with the full diff as clipboard content plus the evidence summary as a text block.
- Branch name conflict: append `-2`, `-3` etc. Surface branch name in dialog before confirmation.

---

### Stage 8: Review and Merge

**Inputs:** GitHub PR (created in Stage 7). PR comment with deep link.

**Outputs:** PR merge (human action). Standard Terraform apply pipeline runs on merge (outside agent scope).

**Services:** GitHub (standard PR review), HCP Terraform (apply pipeline on merge - standard, not agent-initiated)

**Validations:** Human approval. No system guard required at this stage.

**Session link resolution:** `GET /workbench/{session_id}` returns session data from HCP Terraform. Must return 200 with session context if session is within 30-day retention window. Must return 404 with a readable message after expiry.

### Stage 9: Audit

**Inputs:**
- Query by any of: `session_id`, `run_id`, `workspace_id`, `user_identity`, `date_range`

**Outputs (full audit record):**
```json
{
  "session_id": "string",
  "run_id": "string",
  "workspace_id": "string",
  "triggered_by": "string (user email or identity)",
  "triggered_at": "ISO 8601 timestamp",
  "mode": "string",
  "trust_level": "string",
  "intent": "string",
  "context_used": {
    "artifacts": [
      { "type": "run | variable_set | module | policy_set | vcs_file", "id": "string" }
    ]
  },
  "tools_called": [
    { "tool": "string", "action": "read | write", "artifact": "string", "timestamp": "ISO 8601" }
  ],
  "failure_classification": "string",
  "confidence_label": "string",
  "proposed_diff": "{ file_path, before, after }",
  "user_modified_diff": "boolean",
  "modified_diff": "{ file_path, before, after } | null",
  "speculative_run_id": "string | null",
  "speculative_plan_result": "string | null",
  "policy_result": "string | null",
  "pr_approval_action": {
    "approved_by": "string",
    "timestamp": "ISO 8601"
  },
  "pr_url": "string | null",
  "pr_merged_by": "string | null",
  "pr_merged_at": "ISO 8601 | null",
  "terminal_stage": "integer (1-9)",
  "terminal_action": "string"
}
```

**Services:** HCP Terraform audit log (read-only, existing event store extended with agent session fields)

**Validations:**
- Audit record must be immutable. No delete, update, or overwrite operation is permitted after write.
- Audit record must be created at session start (Stage 2) and continuously updated through the session. It must not depend on session completion to be written.
- Audit record must be searchable by `run_id`, `workspace_id`, `session_id`, and `triggered_by` (user identity).
- Audit record retention: minimum 90 days [A: confirm with compliance requirements].

## State Machine

Agent session states and allowed transitions for `session_id`:

| State | Description | Valid next states |
|---|---|---|
| `triggered` | Manual webhook fired; AGX is assembling context | `diagnosis_ready`, `failed` |
| `diagnosis_ready` | Agent returned structured diagnosis to UI | `workbench_open`, `terminated` |
| `workbench_open` | User opened workbench; fix diff is loaded | `fix_under_review`, `terminated` |
| `fix_under_review` | User is reviewing proposed diff | `validation_requested`, `terminated` |
| `validation_requested` | Speculative run has been triggered | `validation_complete`, `validation_failed` |
| `validation_complete` | Speculative plan passed and policy passed | `pr_creation_pending`, `terminated` |
| `validation_failed` | Speculative plan failed or policy failed | `fix_under_review` (user may edit diff and retry), `terminated` |
| `pr_creation_pending` | User has clicked "Create PR" and confirmed | `pr_created`, `pr_creation_failed` |
| `pr_created` | GitHub PR and PR comment have been posted | `merged` (external event), `terminated` |
| `merged` | PR has been merged (external event from GitHub webhook) | terminal |
| `terminated` | Session ended by user at any stage | terminal |
| `failed` | System failure at any stage (AGX unreachable, API error) | terminal |

**Allowed transitions only:**
- `validation_failed` → `fix_under_review`: user may edit the diff and re-validate. This creates a new speculative run linked to the same session.
- `pr_created` → `merged`: external trigger (GitHub webhook or polling). The session record is updated; no user action in HCP Terraform is required.
- Any state → `terminated`: user explicitly closes session. Audit record records `terminal_stage` and `terminal_action`.
- Any state → `failed`: system failure. Audit record records `terminal_stage: N`, `terminal_action: "system_failure"`, `failure_reason: string`.

**Invalid transitions:**
- `validation_requested` → `pr_creation_pending`: speculative run must complete first.
- `diagnosis_ready` → `pr_created`: PR creation requires workbench, fix review, and validation.
- Any terminal state → any other state: sessions cannot be re-activated. A new session must be started.

## Service Dependency Map

| Stage | HCP Terraform data store | AGX | Terraform MCP Server | GitHub VCS connector | HCP Terraform API | Policy engine |
|---|---|---|---|---|---|---|
| 1 | Read (run data) | - | - | - | - | - |
| 2 | - | Receive trigger | Read (context assembly start) | - | - | - |
| 3 | - | Return diagnosis | Read (plan log, variable set, module, run history) | - | - | - |
| 4 | - | Return workbench payload | Read (variable set) | Read (current file) [gated] | - | - |
| 5 | - | - | - | - | - | - |
| 6 | - | Interpret result | - | - | Trigger speculative run (write) | Invoked by speculative run |
| 7 | - | Generate PR content | - | Write (branch, PR, comment) | - | - |
| 8 | - | - | - | Read (PR review) | Apply pipeline (post-merge) | - |
| 9 | Read (audit log) | - | - | - | - | - |

**Dependency notes:**
- Stage 4 GitHub read (current file) is gated on AGX Priority 4 (authenticated connectors). If unavailable, Stage 4 proceeds with `diff_source: variable_set_analysis`.
- Stage 7 GitHub write requires VCS-connected workspace AND authenticated GitHub connector. If VCS not connected, "Copy diff" fallback activates.
- Stages 1-5 are entirely read-only. The first write action is in Stage 6 (speculative run, which is a controlled HCP Terraform operation not a production change). The second write action is in Stage 7 (GitHub PR creation), which requires explicit user approval.

## Timing Budget

| Stage | Target duration | Notes |
|---|---|---|
| Stage 2 trigger acknowledgment | < 1 second | UI confirms button click; spinner activates |
| Stage 3 diagnosis response | < 10 seconds [A] | From button click to inline panel rendered. Target from Meridian PRD MER-16. |
| Stage 3 diagnosis (large plan logs, >500 lines) | < 20 seconds [A] | Agent processes relevant error section only |
| Stage 4 workbench load | < 2 seconds [A] | Pre-populated; agent assembles diff in parallel with Stage 3 |
| Stage 6 speculative run | 30 seconds to 3 minutes [A] | Depends on workspace plan complexity; progress state must be visible |
| Stage 7 PR creation | < 5 seconds [A] | Once user confirms; GitHub API call |
| Stage 9 audit query | < 2 seconds [A] | Read from indexed audit log |

**[A]** = assumed; not confirmed by engineering. All timing targets require validation against actual AGX and HCP Terraform API performance.

**Loading state requirements:**
- Stage 3 (diagnosis): spinner with countdown from `estimated_response_seconds`. If response exceeds estimate by 50%, update countdown. If response does not arrive within 30 seconds, surface error state.
- Stage 6 (speculative run): progress bar or spinner with elapsed time. If run exceeds 5 minutes, surface "This is taking longer than expected. The speculative run may be queued."

## Bridging Data Contract

Values that must be consistent across all system boundaries in a single session. Inconsistency at any boundary is a bug.

| Value | Written by | Read by | Constraint |
|---|---|---|---|
| `session_id` | UI (Stage 2) | AGX (all stages), HCP TF audit log (all stages), GitHub PR comment (Stage 7) | Must be identical across all references. Never regenerated mid-session. |
| `run_id` | HCP Terraform (Stage 1) | AGX (Stage 2-3), HCP TF API (Stage 6), HCP TF audit log (all stages) | Must reference the same run throughout. |
| `workspace_id` | HCP Terraform (Stage 1) | AGX (Stage 2-4), HCP TF API (Stage 6), HCP TF audit log (all stages) | Must reference the same workspace throughout. |
| `proposed_diff` | AGX (Stage 4) | UI workbench (Stage 5), HCP TF API speculative run (Stage 6), GitHub VCS connector (Stage 7), HCP TF audit log | If user modifies diff in Stage 5, both `original_diff` and `modified_diff` must be preserved. The diff applied in Stage 6 must match the diff applied in Stage 7. |
| `speculative_run_id` | HCP Terraform API (Stage 6) | AGX interpretation (Stage 6), HCP TF audit log (Stage 9) | Links validation result to the specific speculative run. |
| `ready_for_pr` | AGX (Stage 6) | UI workbench (Stage 7 gate) | Must reflect both `speculative_plan_result == passed` AND `policy_result == passed`. Must be recalculated if the diff changes between validation attempts. |
| Evidence citations | AGX (Stage 3) | UI evidence panel (Stages 3-5), PR description (Stage 7), HCP TF audit log (Stage 9) | Citation text must be exact excerpts, not paraphrased. Must be identical in the workbench and the PR description. |

## Open Questions

- Is the audit record written to the existing HCP Terraform run/workspace event log (appended fields), or to a new agent session store? The current PDR assumes the existing log is extended. If a new store is required, retention, indexing, and query API need separate specification. -- Engineering
- What is the schema for the existing run event log? PDR-003 assumes appending `session_id`, `tools_called`, `failure_classification`, and related fields. If the existing schema does not support this, a migration or a separate log is needed. -- Engineering
- Does the speculative run in Stage 6 need to be triggered by AGX via MCP, or directly by the UI via the HCP Terraform API? The current contract assumes UI triggers the speculative run via the HCP Terraform API. If AGX triggers it via MCP, the timing and session linking need adjustment. -- AGX team
- What is the confirmed retention period for agent session records? The bridging data contract requires session records to outlive the speculative run. PDR assumes 30-day minimum for workbench deep link; 90-day minimum for audit record. These need compliance confirmation. -- Compliance / Engineering
- Is `ready_for_pr` computed by AGX or by the UI? If the UI computes it from the speculative run result fields, AGX does not need to send it explicitly. If AGX sends it, it is a hard system dependency. Current PDR assumes AGX sends it. -- Engineering
- When the user edits the diff in Stage 5 and then re-validates in Stage 6, does a new `speculative_run_id` replace the old one, or are both retained in the audit record? The current contract retains both: `speculative_run_id` is the latest run, and audit log records all validation attempts. -- Engineering
