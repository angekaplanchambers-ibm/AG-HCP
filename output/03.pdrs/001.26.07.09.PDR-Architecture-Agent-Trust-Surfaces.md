## Problem Statement

When a platform engineer or app developer encounters a failed run, drift, or upgrade risk in HCP Terraform, they have no way to get a grounded explanation, a reviewable fix, or a validated change without manual log triage. The platform has all the relevant data (run logs, plan output, policy results, workspace state, module sources) but provides no reasoning layer over it.

Introducing an agent creates a new set of questions the architecture must answer before any surface is designed: how much authority does the agent have, when does it require human approval, how does the user know what the agent is doing, and how does an auditor verify what happened? This PDR answers those questions as a shared contract for all downstream design and engineering work.

---

## Design Pattern

**Verified intent loop.** The system translates a user's stated intent (or a detected failure event) into a grounded, evidence-backed action proposal that the user must explicitly approve before any write action occurs. No production change happens without a human decision. The agent's role is to reduce the effort of making a well-informed decision, not to eliminate the decision.

This pattern applies across all six CUJs in strategy doc 002. Every CUJ follows the same loop:

```
Trigger (user action or event)
    |
    v
Context assembly (agent reads Terraform-native data)
    |
    v
Diagnosis / analysis (agent classifies, cites evidence, proposes)
    |
    v
Human review (user reviews evidence and proposed action)
    |
    v
Validation (speculative run, policy check, or blast-radius analysis)
    |
    v
Approval gate (explicit user decision before any write action)
    |
    v
Execution (agent acts within approved scope)
    |
    v
Audit record (immutable log of intent, context, decision, outcome)
```

The loop is not completed automatically. The user may exit at any stage. Every exit point must be a clean terminal state: no partial writes, no dangling agent sessions.

## Trust Ladder

The Trust Ladder defines what the agent is authorized to do. It is the primary governance mechanism. It is set per organization and optionally overridden per workspace. Sprint 4 Option B consensus (source: `017`) confirms L2 as the correct MVP target.

| Level | Label | What the agent does | What requires human approval | Status |
|---|---|---|---|---|
| L1 | Explain only | Classify failure, cite evidence, return structured diagnosis | Everything - no write actions, no proposals | Design target for first-run regulated customers |
| L2 | Propose and validate | Explain + generate proposed diff + trigger speculative run + open PR draft | Merge and apply - agent never applies | **MVP target - confirmed by Sprint 4 Option B consensus** |
| L3 | Auto-close clean plans | As L2, but closes PRs automatically when speculative plan is clean with no risk flags | Apply only | Near-future target (Option C in Sprint 4 survey) |
| L4 | Autonomous operate | Full autonomous execution including apply | None within defined scope | Explicitly out of scope for MVP. Requires separate governance review. |

L1 and L2 must both be available at launch. Regulated-environment customers (BFSI, healthcare, government) require L1 as their initial trust level before incrementally adopting L2. Source: Sprint data across all four research sprints; `017` Theme 2 and Theme 7.

The active Trust Ladder level must be visible to the user at all times while an agent session is active. Mode indicator design must reflect the current level.

## Agent Modes

Four modes correspond to the operations an agent can perform within a session. Mode is separate from Trust Ladder level: the Trust Ladder defines what the organization allows; mode is what the user selects for a specific session within those limits.

| Mode | What the agent does | Requires user approval? | AGX trigger needed |
|---|---|---|---|
| Explain | Read context, classify, cite evidence, return structured diagnosis | No - read only, no write actions | Manual webhook |
| Plan | Explain + generate proposed diff + project speculative plan result | No - no write actions yet | Manual webhook |
| Prepare PR | Plan + trigger speculative run + create GitHub PR with evidence | Yes - user approves PR creation step | Manual webhook + GitHub VCS connector |
| Operate | [Out of MVP scope] | - | - |

Default mode for the "Ask Terraform" button: Explain. Users must explicitly escalate to Plan or Prepare PR. Mode transitions require explicit user action and must not happen silently.

## Surface Inventory

Eight surfaces. Four are new. Four extend existing HCP Terraform surfaces.

| Surface | Type | Entry point | Agent role | Approval required? |
|---|---|---|---|---|
| Run page entry point | New | Failed run page | Trigger: user clicks "Ask Terraform" | No |
| Inline explanation panel | New | Run page (expands in place) | Renders diagnosis: failure class, evidence, confidence label, mode | No |
| Workbench | New | From inline panel or Slack pulse | Evidence panel, proposed fix diff, speculative run result, approval controls | Yes - PR creation |
| Approval gate | New | Within workbench | Surfaces write-action boundary; what the agent is requesting to do | Yes |
| PR comment | Extends GitHub PR | GitHub PR (created by agent) | Structured diagnosis, evidence link, validation status, next action | No (created after user approves PR creation) |
| Slack pulse | Extends Slack | Scheduled / event-driven | Estate health summary: failed runs, drift, policy violations, upgrade risks | No (Slack actions route to workbench for approval) |
| MCP scope panel | New | Wherever MCP session is active | Read/write scope, enabled tools, token identity, approval boundaries | No (configuration, not action) |
| Admin audit view | Extends HCP Terraform workspace/run history | Admin workspace settings | Agent session trace: intent, context used, tools called, diff, validation, approval | No (read only) |

## Journey Flow

User movement across surfaces follows the verified intent loop. CUJ-01 (failed-run repair) is specified in full in journey map `005`. The pattern generalizes to all CUJs.

```
[Run fails] or [Slack pulse fires] or [Upgrade risk detected]
    |
    v
Run page or Slack pulse
    |
    v  (user clicks "Ask Terraform" or opens workbench from Slack)
Inline explanation panel  [Explain mode - read only]
    |
    v  (user opens workbench)
Workbench  [Plan mode - diff + speculative projection]
    |
    v  (user triggers speculative run)
Validation result in workbench
    |
    v  (speculative pass + policy clear -> user approves PR creation)
Approval gate
    |
    v  (user approves)
GitHub PR + PR comment created
    |
    v  (approver reviews)
Merge and apply  [human action, outside agent scope]
    |
    v  (throughout)
Admin audit view  [agent session trace, available to admins at any stage]
```

Cross-surface continuity is maintained by a session ID that links the run page diagnosis to the workbench to the PR comment. The session ID is the connective tissue across all surfaces. Session records are retained for a minimum of 30 days [A].

## Editability Model

| Item | Editable by user? | When | How it propagates |
|---|---|---|---|
| Agent mode for this session | Yes | Before or during session | Mode indicator updates; session re-evaluates from selected mode |
| Trust Ladder level for the organization | Yes (admin only) | In org or workspace settings | All new sessions inherit the new level; active sessions are not interrupted |
| Proposed fix diff | Yes | In workbench, before speculative run | Edited diff triggers a new speculative run on next validation click |
| PR title and description | Yes | In PR creation dialog, before confirming | Applies to the GitHub PR; does not affect the HCP Terraform audit record |
| Evidence panel (agent's cited sources) | No | - | Read only; user cannot remove citations (audit integrity requirement) |
| Speculative run result | No | - | System output; read only |
| Audit record | No | - | Immutable. No user or admin action can modify or delete it. |
| MCP scope (enabled tools, token) | Yes (admin only) | In MCP scope panel | New scope applies to the next session start; active sessions use scope from session start |

## Agent Responsibilities vs. User Decisions

| Action | Agent | User |
|---|---|---|
| Read run data, plan log, policy result, workspace context | Automatic (on trigger) | No action needed |
| Classify the failure | Automatic | User reviews and can reject classification |
| Cite specific evidence | Automatic | User verifies evidence in evidence panel |
| Generate proposed fix diff | Automatic | User reviews diff; may edit before validation |
| Trigger speculative run | On user approval | User clicks "Validate" |
| Interpret speculative run result | Automatic | User reviews result and decides whether to proceed |
| Create GitHub branch and PR | On user approval | User clicks "Create PR" and confirms |
| Post PR comment | Automatic (on PR creation approval) | No action needed |
| Record audit trail | Automatic (continuous throughout session) | No action needed |
| Apply changes | Never - not in scope | Approver merges PR; infrastructure applies through standard pipeline |

Hard constraints:
- The agent never applies changes to production infrastructure.
- The agent never creates a PR without explicit user approval.
- The agent never operates above the Trust Ladder level set for the organization.

## Rollback Model

| Action | Reversible? | How | User sees |
|---|---|---|---|
| Agent session triggered (Explain or Plan mode) | Yes - trivially | Close session; no state changed | Session closes; no artifact created |
| Speculative run triggered | Partial | Speculative runs do not change state; they are read-only validation | Result stored in session; closing workbench discards the session result |
| PR created | Yes - in GitHub | User or approver closes or reverts the PR in GitHub | PR closed; audit record notes PR was created and closed |
| PR merged and changes applied | Via standard Terraform revert | User creates a new PR reverting the change; applies via standard pipeline | New run, new PR; outside agent scope |

The agent has no revert mechanism for applied infrastructure changes. Revert is a user action through the standard Terraform workflow. The audit trail provides the evidence needed to determine what to revert and why.

## Specialist Review Hooks

Four points where async specialist review can be injected:

| Hook | Stage (per `005`) | Who reviews | What they see |
|---|---|---|---|
| Evidence review | Stage 5 (proposed fix review) | Platform engineer, senior engineer, SRE | Full evidence panel, proposed diff, confidence label, uncertainty notes |
| Policy specialist review | Stage 6 (validation) - if policy fails | Policy engineer, security reviewer | Policy failure name, rule text, cited resource, proposed fix and why it may not clear the policy |
| PR review | Stage 8 (review and merge) | Any authorized approver | PR comment with diagnosis and evidence link, full PR diff, optional workbench deep link |
| Admin audit review | Stage 9 (audit) | Security, compliance, or audit team | Full agent session trace: intent, context, tools called, diff, validation result, approval chain |

For regulated environments: the evidence panel is the mechanism that makes the evidence review hook functional. A reviewer must be able to verify or reject the agent's reasoning using only what is surfaced in the evidence panel, without needing access to raw run logs or internal AGX state. This is a hard design requirement, not a nice-to-have.

## AGX Trigger and Response Boundary

This architecture operates across two layers (strategy doc `003`). The trigger/response boundary is the seam design must specify before implementation. The contract below is the minimum required for PDR-002 and PDR-003 to proceed.

### Trigger: UI → AGX

| Trigger type | Status (as of 2026-07) | Surface unlocked |
|---|---|---|
| Manual webhook | Desired state - AGX Priority 3 | Run page "Ask Terraform" button, workbench invocation |
| Run/native task webhook | Desired state - AGX Priority 3 | Auto-invocation on run failure (auto-notification to user) |
| Notification webhook | Desired state - AGX Priority 3 | Slack pulse event-driven posting |
| Scheduled trigger | Desired state - AGX Priority 3 | Daily infra pulse |

Minimum trigger payload (manual webhook):

| Field | Type | Description |
|---|---|---|
| `run_id` | string | HCP Terraform run identifier |
| `workspace_id` | string | HCP Terraform workspace identifier |
| `user_identity` | token reference | MCP passthrough; establishes who triggered and what scope they have |
| `mode` | enum | `explain` \| `plan` \| `prepare_pr` |
| `session_id` | string | UI-generated; used to link run page, workbench, PR comment, and audit record |

### Response: AGX → UI

Minimum response payload:

| Field | Type | Description |
|---|---|---|
| `session_id` | string | Echoed from trigger for cross-surface continuity |
| `failure_classification` | enum | `missing_variable` \| `invalid_module_input` \| `provider_version_conflict` \| `policy_failure` \| `drift_surprise` \| `unclassified` |
| `evidence_citations` | array | Each item: `{ type, source, reference, excerpt }` |
| `confidence_label` | enum | `high` \| `medium` \| `low` |
| `confidence_rationale` | string | Brief explanation of why confidence is at this level |
| `uncertainty_notes` | string[] | What the agent could not determine (named gaps, not vague disclaimers) |
| `proposed_fix_diff` | diff object or null | `{ file_path, before, after }` per changed file; null in Explain mode |
| `agent_mode` | enum | Active mode for this response |
| `tool_scope_used` | array | Each item: `{ tool_name, action, artifact_reference }` |
| `workspace_context` | object | `{ variable_set_id, module_sources[], vcs_repo, last_successful_run_id }` |

Validation result payload (returned after speculative run):

| Field | Type | Description |
|---|---|---|
| `speculative_run_id` | string | HCP Terraform speculative run identifier |
| `speculative_plan_result` | enum | `passed` \| `failed` |
| `resources_changed` | number | Count of planned resource changes |
| `policy_result` | enum | `passed` \| `failed` |
| `failing_policy_name` | string or null | Name of failing policy if result is failed |
| `updated_confidence_label` | enum or null | Revised confidence after seeing speculative result |
| `ready_for_pr` | boolean | True only if speculative plan passed AND policy passed |

`ready_for_pr: false` must block the PR creation path in the UI. This is a hard guard, not a soft warning.

## Scope

### In scope for this PDR

- Trust Ladder definition, level constraints, and MVP target
- Agent mode definitions and transition rules
- Surface inventory for MVP (eight surfaces)
- Journey flow across surfaces (CUJ-01 detail in `005`)
- Editability model
- Agent vs. user responsibility boundary
- Rollback model
- Specialist review hooks
- Trigger and response data contracts at the AGX boundary

### Out of scope for this PDR

- AGX internal reasoning, model selection, context retrieval, or orchestration (AGX layer - see `003`)
- Per-CUJ scenario scripts (PDR-002)
- Per-stage data contracts with validation and state machine detail (PDR-003)
- Wireframe specifications (PDR-004)
- Meridian Z-specific agents and workflows
- AGX multi-tenancy, BYOP, or execution scalability implementation
- CUJ-05 (upgrade lifecycle) and CUJ-06 (blast-radius) detailed journeys - these follow the same verified intent loop pattern but require separate journey maps

## Open Questions

- What is Project Solis? Who owns it, and what does it deliver as the Terraform Commercial conversational UI? The Meridian vision doc (`019`) states Meridian integrates into Terraform Commercial UIs using Project Solis as a micro-front-end. If Solis provides the workbench chat interface out of the box, PDR-004's workbench wireframe spec needs to be reconciled with what Solis ships before build begins. -- Chris Audie / Daniel Savage
- Are AGX (Marc Cosentino's backend runtime) and Meridian's AI Harness / AI Reasoning Loop the same system, converging, or parallel? If converging, the trigger/response architecture in this PDR applies to the merged system. If parallel, there are two competing backends for the same frontend surfaces. Must be confirmed before the trigger/response contract is finalized. -- Chris Audie
- Is the `terraform intent XYZ` CLI command in the six-month window? If yes, it should be added as a surface entry point in the Surface Inventory alongside run page, Slack, GitHub PR comment, and MCP panel. Design can proceed against a mock; confirmation of timeline determines whether wireframes are required before the workshop. -- Daniel Savage / Omar Ismail
- Who owns the trigger/response API contract on the AGX side? Without a named owner, the boundary cannot be finalized. -- AGX team / Marc Cosentino
- Does the session ID need to be generated by the UI or by AGX? If AGX generates it, the UI cannot create the PR comment deep link until the agent's first response. -- Engineering
- Is Trust Ladder level an organization-level setting, workspace-level override, or both? Current design assumes org-level with optional workspace override. -- PM (Omar Ismail / Daniel Savage)
- What does the user see when failure classification returns "unclassified"? If the agent cannot classify the failure, what actions remain available? -- Design
- For regulated customers requiring L1 at first: is Trust Ladder level configurable without engineering involvement, or is it a provisioning-time setting? -- PM / Engineering
- Does the manual webhook trigger support passing agent mode at invocation, or does mode selection happen client-side after the agent's first response? -- AGX team
- Is the `evidence_citations` array format sufficient to drive the evidence panel UI, or does the panel require structured field types (e.g., distinguishing a log-line citation from a policy-rule citation)? -- Design + Engineering
