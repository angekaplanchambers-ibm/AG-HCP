## Problem Statement

PDRs 001-003 define the architecture, scenario, and data contracts. This PDR specifies what to build in Storybook: which surfaces, which states per surface, what each story must demonstrate, what fixture data drives them, and what the QA checklist verifies before stories ship. It is the design-to-implementation handoff for the wireframe phase.

---

## Design Constraints

**Fidelity:** Wireframes are low fidelity and gray until design direction is approved. No brand color, no final typography, no production-ready styling. Layout, information hierarchy, and interaction states are what matters at this stage.

**Surfaces:** Web-only for the run page, workbench, MCP scope panel, and admin audit view. GitHub PR comment is rendered as a static mockup (not interactive). Slack pulse is rendered as a static mockup with action buttons annotated.

**Component approach:** Use unstyled layout blocks or a minimal gray design token set. No HashiCorp Design System components at this stage - the goal is design logic, not visual polish.

**Accessibility:** Every interactive element must be reachable by keyboard. Focus order must match visual reading order. Labels must be present on all form elements and buttons. Screen reader annotations are not required at wireframe stage but element labels must be meaningful.

**Responsive:** Desktop-primary. Workbench and run page are desktop surfaces. No mobile layout required at this stage.

**Motion and timing:** Loading states must be represented as static wireframes (spinner placeholder, skeleton state). Actual animation is not in scope for wireframes.

**Truncation and overflow:** All text fields must have defined overflow behavior (truncate, expand, scroll). Do not wireframe with perfectly fitting text; use realistic-length fixture data.

## Story Specifications

### RunPage.stories.tsx

**Purpose:** Show the run page entry point across four states: no agent (baseline), agent entry visible, agent working, and inline explanation returned.

---

**Story 1: Baseline**
State: Run is in errored status. No agent action taken. "Ask Terraform" button is visible.

What to show:
- Run header: workspace name, run ID, status badge (`errored`)
- Raw error summary (truncated, expand available)
- "Ask Terraform" button - primary, adjacent to error summary
- Mode selector: default Explain, dropdown for Plan / Prepare PR
- No agent content yet

Fixture: `mockRun` with status `errored`. No session data.

Interactions: Button click transitions to Story 3 state.

---

**Story 2: Non-failed run (no button)**
State: Run is in `applied` status. "Ask Terraform" button must not appear.

What to show:
- Run header with status `applied`
- No "Ask Terraform" button present
- Annotation: "Button appears only when status is errored"

Fixture: `mockRun` with status overridden to `applied`.

Interactions: None.

---

**Story 3: Agent working**
State: User has clicked "Ask Terraform." Trigger acknowledged. Spinner visible.

What to show:
- Mode indicator: "Agent working - Explain mode"
- Spinner with countdown: "Estimated response: 8 seconds"
- "Ask Terraform" button replaced by mode indicator (button is not clickable while agent is working)
- Inline explanation panel area: empty skeleton

Fixture: `mockSession` with mode `explain`. No diagnosis yet.

Interactions: Countdown shows. Transitions to Story 4 on response arrival.

---

**Story 4: Explanation ready**
State: Agent returned structured diagnosis. Inline explanation panel is populated.

What to show:
- Mode indicator: "Explain mode - no changes proposed yet"
- Inline explanation panel expanded below error summary (see InlineExplanationPanel Story 1 for panel spec)
- "Open workbench" button available

Fixture: `mockRun`, `mockSession`, `mockDiagnosis`.

Interactions: "Open workbench" navigates to Workbench Story 1.

### Workbench.stories.tsx

**Purpose:** Show the workbench across five key states covering the full Stage 4-7 flow.

Layout: Two-column. Left/top: evidence panel. Right/main: proposed fix diff, speculative run result, approval controls.

---

**Story 1: Pre-validation**
State: Workbench loaded with evidence panel and proposed fix diff. Speculative run not yet triggered.

What to show:
- Evidence panel: three citations, confidence label, uncertainty notes, context chips
- Proposed fix diff: file path, before/after with line highlights
- Diff source note if applicable (estimated from variable set analysis)
- Speculative plan projection: "Expected: 1 resource update (aws_db_instance.main)" [projected, not validated]
- Policy prediction: "Expected to pass payments-policy"
- "Validate with speculative run" button (primary)
- Mode indicator: "Prepare PR mode"
- Trust ladder level: "L2 - agent will propose; you approve"

Fixture: `mockDiagnosis`, `mockProposedDiff`, `mockSession`.

---

**Story 2: Validation in progress**
State: User has clicked "Validate." Speculative run is executing.

What to show:
- Evidence panel: same as Story 1 (no change)
- Diff: same (greyed out / locked during validation)
- Validation section: spinner with elapsed time "Speculative run in progress... 0:15"
- "Validate" button disabled during run
- Mode indicator unchanged

Fixture: `mockDiagnosis`, `mockProposedDiff`, `mockSession`. Validation state: `loading`.

---

**Story 3: Validation passed - ready for PR**
State: Speculative plan passed. Policy passed. PR creation available.

What to show:
- Evidence panel: unchanged
- Validation result section: green status "Speculative plan passed - 1 resource update" and "Policy passed"
- "Create PR" button (primary, enabled)
- "Validate" button replaced by validation result
- Mode indicator: "Prepare PR mode - validation complete"

Fixture: `mockDiagnosis`, `mockProposedDiff`, `mockSession`, `mockValidationResult`.

---

**Story 4: Validation failed - re-diagnose**
State: Speculative plan failed. Agent re-diagnoses the delta failure.

What to show:
- Validation result section: red status "Speculative plan failed"
- Updated delta: "Policy 'require-approved-modules' is still failing: resource aws_db_instance.main uses an unapproved module version."
- Evidence panel updated with new citation (policy failure from speculative run)
- Diff is editable again
- "Validate" button re-enabled
- "Create PR" button remains disabled (not ready for PR)

Fixture: Custom fixture with `speculativePlanResult: "failed"`, `policyResult: "failed"`, `failingPolicyName: "require-approved-modules"`, `readyForPr: false`.

---

**Story 5: VCS not connected**
State: Workspace is not connected to a VCS repo. PR creation path unavailable.

What to show:
- Evidence panel: same
- Diff: shown (estimated from variable set analysis)
- Diff source note: "This workspace is not connected to a VCS repository. The diff below is estimated from variable set analysis. PR creation is not available for this workspace."
- "Copy diff to clipboard" button (primary)
- PR creation section: not visible
- "Validate with speculative run" still available

Fixture: `mockDiagnosis`, custom diff with `diffSource: "variable_set_analysis"`, no VCS repo in workspace context.

## Storybook Conventions

**Naming:** All stories use CUJ-prefixed titles. Format: `[Surface] / [State]`. Example: `Workbench / Validation passed - ready for PR`.

**Organization:** All wireframe stories in `storybook/stories/wireframes/`. Stories are flat within this folder; no subdirectory nesting.

**File format:** CSF3. Each story exports a default meta and named story objects.

**Fixture imports:** Each story file imports from its co-located `_*-fixtures.ts` file. Shared fixtures import from `_shared-fixtures.ts`.

**Args:** Use Storybook args for all configurable state (mode, confidence level, validation result, credential type). Annotate with JSDoc for each arg.

**Story names must match PDR-004 story names exactly.** The QA checklist references stories by name; name drift creates gaps.

## QA Checklist

Before stories ship to the showcase build:

**Coverage:**
- [ ] All 26 stories render without errors
- [ ] Every story name from the file manifest exists in Storybook

**Data integrity:**
- [ ] All evidence citations show actual excerpts (not placeholder text)
- [ ] Confidence labels match the fixture data (not hardcoded in component)
- [ ] Diff file paths and before/after content are realistic (no lorem ipsum)
- [ ] Session ID appears consistently across all surfaces in linked stories

**Interaction states:**
- [ ] "Ask Terraform" button is not visible in RunPage Story 2 (applied run)
- [ ] "Create PR" button is disabled in Workbench Story 4 (validation failed)
- [ ] "Create PR" button is disabled in Workbench Story 5 (VCS not connected)
- [ ] Approval gate is not reachable unless `readyForPr: true`

**Audit completeness:**
- [ ] AdminAuditView Story 1 shows all 9 stages (some as tool calls, not user events)
- [ ] AdminAuditView Story 2 shows terminal_stage: 5, no write actions
- [ ] AdminAuditView Story 3 shows no diagnosis, no write actions

**Accessibility baseline:**
- [ ] All buttons have visible labels (no icon-only buttons without aria-label)
- [ ] Evidence citations are readable in the context of the panel (not just tooltips)
- [ ] Mode indicator is a visible, persistent element in all agent-active stories

**PR comment:**
- [ ] Deep link is present in Stories 1 and 3 (not present in Story 2 - unclassified)
- [ ] Story 3 correctly shows "PR not created" and blocks the implication that a PR was opened

**Slack pulse:**
- [ ] Action buttons in Story 2 have annotation labels explaining the scope of each action
- [ ] "All write actions happen in HCP Terraform, not in Slack" note is visible

## Open Questions

- What does the workbench host surface look like in the current HCP Terraform UI? Is it a new page, a modal over the run page, or a slide-over panel? The wireframe plan assumes a new page (full-screen workbench). If it is a modal or panel, the layout constraints change significantly. -- Design + HCP TF UI team
- Does the GitHub PR comment support markdown rendering for the evidence block, or is it plain text only? The PR comment stories assume markdown. -- Engineering
- Is the Slack pulse a Slack Block Kit message (supports action buttons) or a plain webhook notification? The Slack pulse stories assume Block Kit. If it is a plain webhook, action buttons are not available and the surface is information-only. -- Engineering
- Are there existing HCP Terraform UI components for status badges, timeline views, or diff rendering that should be used as the structural basis for wireframes? Using existing components avoids redundancy. -- Design + HCP TF UI team
- For the admin audit view, is the timeline view the right structure, or does the team prefer a tabular log format? The timeline is more readable for a single session; tabular works better for cross-session queries. The current stories only cover single-session views. -- Design + PM
- Does the MCP scope panel live as a persistent panel visible during all agent sessions, or is it a modal accessible via an "Agent scope" link? The current stories assume a panel accessible via a link/button, not always visible. -- Design
