## Problem Statement

When a run fails in HCP Terraform, the platform engineer has the data needed to diagnose the failure (run log, plan output, policy results, workspace state) but no system to interpret it. They spend 10-30 minutes on manual log triage for failures that recur weekly. For regulated-environment operators, every diagnostic step must be traceable to an evidence source. This scenario specifies the exact sequence of actions, system calls, and data payloads that implement CUJ-01 from trigger to audit record.

## Environment

**System state at scenario start:**

- HCP Terraform workspace `payments-prod` is connected to a GitHub VCS repo (`org/infra-monorepo`), branch `main`
- Run `run-abc123` has completed with status `errored`
- Run log contains: `Error: Invalid value for variable "db_instance_class" on .terraform/modules/rds/main.tf line 34`
- Policy result: passed (no policy failures in this scenario)
- Variable set `payments-prod-vars` is attached; variable `db_instance_class` is not set
- Module: `registry.terraform.io/hashicorp/rds/aws` version `3.5.1`
- Last successful run: `run-xyz789` (3 days ago)
- User: `alex@example.com`, Trust Ladder level for org: L2 (Propose and validate)
- Terraform MCP Server: configured with read scope on workspace `payments-prod`; write scope includes create branch and open PR
- AGX: manual webhook URL is registered for the workspace [A: desired state - AGX Priority 3]
- Session ID: `sess-001` (generated by UI at button click)

**Preconditions:**

- Run `run-abc123` is in `errored` state
- User has Terraform workspace read permission
- VCS connection is active (authenticated GitHub connector) [A: AGX Priority 4 for write; read may be available earlier]
- AGX manual webhook is reachable

## Stage-by-Stage Script

### Stage 1: Trigger

**Operator:** Alex lands on the run page for `run-abc123` in HCP Terraform (via email notification or direct navigation). The run page shows status `errored` and displays the raw error summary: `Invalid value for variable "db_instance_class"`.

**Agent:** None. Run has already completed. All run data is at rest in HCP Terraform.

**Data:**
- Available in HCP Terraform run record: `run_id=run-abc123`, `workspace_id=ws-payments-prod`, `status=errored`, `plan_log_url`, `policy_result=passed`, `vcs_commit_sha`, `linked_workspace_vars=payments-prod-vars`
- No AGX call in this stage.

---

### Stage 2: Request

**Operator:** Alex clicks "Ask Terraform" on the failed run page. A mode selector appears defaulting to "Explain." Alex confirms (or leaves default) and the button fires.

**Agent:** UI generates `session_id=sess-001`. UI sends manual webhook trigger to AGX.

**Data:**

Trigger payload (UI → AGX):
```json
{
  "session_id": "sess-001",
  "run_id": "run-abc123",
  "workspace_id": "ws-payments-prod",
  "user_identity": "<mcp-token-alex>",
  "mode": "explain",
  "trust_level": "L2"
}
```

AGX acknowledges trigger (AGX → UI):
```json
{
  "session_id": "sess-001",
  "status": "processing",
  "estimated_response_seconds": 8
}
```

UI state: mode indicator shows "Agent working - Explain mode." Spinner visible. No write actions have occurred.

AGX begins context assembly via Terraform MCP Server:
- Read: `run-abc123` plan log (relevant error section)
- Read: policy result for `run-abc123`
- Read: variable set `payments-prod-vars` (which variables are set, which are absent)
- Read: module source `registry.terraform.io/hashicorp/rds/aws` version `3.5.1` - registry metadata
- Read: run history for `ws-payments-prod` - last successful run `run-xyz789`

**Failure modes:**
- AGX webhook unreachable: UI surfaces "Could not reach the agent. Try again or open the run log manually." No spinner left open.
- MCP token expired: MCP scope panel surfaces "Your session token has expired. Re-authenticate to continue." No trigger fires.

---

**Operator:** Alex reads the inline explanation panel that expands below the run error summary. No action required.

**Agent:** Returns structured diagnosis to UI.

**Data:**

Agent response payload (AGX → UI):
```json

  "evidence_citations": [
    {
```

Inline panel shows:
- Failure class: "Missing required variable"
- Evidence: three cited sources (log line, variable set absence, module declaration)
- Confidence: High
- Uncertainty: None
- Mode: "Explain - no changes proposed yet"
- Action available: "Open workbench"

**Failure modes:**
- Agent cannot classify failure (returns `unclassified`): panel shows raw error excerpt plus "The agent could not classify this failure type. Review the plan log for manual investigation." No proposed fix surfaced.
- Agent returns low confidence: confidence label shows "Low" with rationale; uncertainty notes name the gap. Workbench is available but fix diff is labeled "Uncertain."

---

### Stage 4: Workbench

**Operator:** Alex clicks "Open workbench." Navigates to the workbench view, pre-populated with session `sess-001` context.

**Agent:** In parallel with returning the Stage 3 diagnosis, the agent (in Plan mode, which the workbench implicitly activates) has begun generating the proposed fix diff. The workbench arrives pre-populated.

**Data:**

AGX generates proposed fix (parallel with diagnosis render):
- Reads current variable set `payments-prod-vars` to confirm which variables are set
- Reads module `rds/aws@3.5.1` input schema to determine valid values for `db_instance_class`
- Generates proposed fix: add `db_instance_class = "db.t3.micro"` to the variable set [A: agent selects smallest valid default value; user must confirm this is appropriate]
- Reads current `main.tf` at the failing path from VCS [A: requires authenticated GitHub connector - AGX Priority 4]

Workbench populated with:
- Evidence panel: full evidence set from Stage 3 (three citations)
- Proposed fix diff:
  ```
  --- a/payments-prod-vars
  +++ b/payments-prod-vars
  @@ -12,0 +13 @@
  + db_instance_class = "db.t3.micro"
  ```
- Speculative plan projection: "Expected: 1 resource update (aws_db_instance.main)"
- Confidence label: High
- Uncertainty note: "Default value 'db.t3.micro' was selected based on module schema. Confirm this is appropriate for the payments-prod environment." [A]
- Context chips: run `run-abc123`, workspace `payments-prod`, variable set `payments-prod-vars`, module `rds/aws@3.5.1`, policy set `payments-policy`
- Action available: "Validate with speculative run"

**Failure modes:**
- VCS connector not available (AGX Priority 4 not yet landed): diff is generated from variable set analysis only; labeled "[Estimated from variable set - file state not verified]"
- Workbench fails to load session `sess-001`: surface "Could not load session. Return to run page and try again." Session data must be retained for the full 30-day window.

---

### Stage 5: Proposed Fix Review

**Operator:** Alex reads the proposed fix diff. Reviews evidence panel. Reads uncertainty note about `db.t3.micro`. Decides the value is appropriate for the environment. Proceeds.

**Agent:** Waiting. No action unless Alex asks a follow-up question. [A: follow-up interaction is post-MVP scope - MLE includes static diff and evidence only]

**Data:** No new system calls. Workbench state is read-only at this stage.

Alex may edit the diff before proceeding:
- If edited: diff is stored as `user_modified_diff` in session; original diff retained in audit record for comparison
- If not edited: diff proceeds as agent-proposed

**Failure modes:**
- Alex cannot trace the fix to the evidence: evidence panel must surface exactly why `db_instance_class` was identified as the problem. If this is not clear, the design has failed - not the system.
- Alex rejects the fix: session terminates. Audit record: `{ session_id: "sess-001", terminal_stage: 5, terminal_action: "user_rejected_fix" }`. Run page returns to errored state with no agent action taken.

---

### Stage 6: Validation (Speculative Run)

**Operator:** Alex clicks "Validate with speculative run." Waits in workbench for the result.

**Agent:** Applies the proposed diff to a temporary workspace variable set configuration, triggers a speculative run via the HCP Terraform API, waits for the result, and interprets it.

**Data:**

Speculative run trigger (Workbench → HCP Terraform API):
```json
{
  "workspace_id": "ws-payments-prod",
  "variables": {
    "db_instance_class": "db.t3.micro"
  },
  "is_speculative": true,
  "message": "Agent-proposed fix validation - session sess-001"
}
```

HCP Terraform runs speculative plan and policy check:
- Plan result: `planned` (1 resource: `aws_db_instance.main` will be updated)
- Policy result: `passed`

Agent interprets and returns validation result (AGX → UI):
```json
{
  "session_id": "sess-001",
  "speculative_run_id": "run-spec-456",
  "speculative_plan_result": "passed",
  "resources_changed": 1,
  "policy_result": "passed",
  "failing_policy_name": null,
  "updated_confidence_label": null,
  "ready_for_pr": true
}
```

Workbench shows:
- "Speculative plan passed - 1 resource planned for update"
- "Policy passed"
- "Ready to create PR"
- Action available: "Create PR"

**Failure modes:**
- Speculative run fails to trigger (API error): surface "Speculative run could not be started. Check workspace status or try again." Do not proceed to PR creation.
- Speculative plan passes but policy still fails: surface "Policy [name] still failing: [rule]." Do not set `ready_for_pr: true`. Agent re-diagnoses the policy failure in context.
- Speculative plan reveals more changes than the proposed fix intended (e.g., 5 resources, not 1): flag to Alex before PR creation: "Speculative plan shows 5 resources changing. Review before proceeding."

**Duration:** Speculative run: typically 30 seconds to 3 minutes depending on workspace size [A].

---

### Stage 7: PR Creation

**Operator:** Alex clicks "Create PR." Reviews the pre-populated PR dialog (title, description, evidence summary). Confirms.

**Agent:** Generates PR title, description, and comment. Creates branch in GitHub. Opens PR. Posts PR comment.

**Data:**

PR creation call (AGX → GitHub VCS connector):
```json
{
  "repo": "org/infra-monorepo",
  "base_branch": "main",
  "new_branch": "terraform-agent/fix-missing-var-run-abc123",
  "diff": {
    "file": "environments/payments-prod/terraform.tfvars",
    "change": "+ db_instance_class = \"db.t3.micro\""
  },
  "pr_title": "[Terraform] Fix missing variable db_instance_class in payments-prod",
  "pr_description": "## Diagnosis\n\nRun run-abc123 failed due to missing required variable `db_instance_class` in variable set `payments-prod-vars`.\n\n## Evidence\n\n- Plan log line 34: `Invalid value for variable \"db_instance_class\"`\n- Variable `db_instance_class` is declared as required in module `rds/aws@3.5.1` with no default\n- Variable is absent from attached variable set `payments-prod-vars`\n\n## Fix\n\nAdds `db_instance_class = \"db.t3.micro\"` to the variable set.\n\n## Validation\n\n- Speculative plan: passed (1 resource update: `aws_db_instance.main`)\n- Policy: passed\n\n[View agent session in HCP Terraform](https://app.terraform.io/workbench/sess-001)",
  "session_id": "sess-001"
}
```

PR comment posted (shorter, for reviewers):
```
**Terraform agent diagnosis - run-abc123**

Failure: Missing required variable `db_instance_class` in `payments-prod-vars`

Fix: Added `db_instance_class = "db.t3.micro"` to variable set

Validation: Speculative plan passed - 1 resource update. Policy passed.

[View full evidence in HCP Terraform workbench](https://app.terraform.io/workbench/sess-001)
```

**System guards (hard):**
- `ready_for_pr` must be `true` before PR creation dialog opens. If false: button is disabled.
- PR creation is blocked if speculative run has not been triggered and completed.

**Failure modes:**
- GitHub branch creation fails (conflict): surface "Branch already exists. Rename or delete the existing branch and try again."
- VCS connection not available (workspace not connected to VCS): surface "This workspace is not connected to a VCS repository. Copy the diff to apply manually." Provide diff clipboard action.
- User edits PR description and removes citations: acceptable; no system guard. Audit record retains original agent-generated content.

---

### Stage 8: Review and Merge

**Operator:** Morgan (PR approver) sees the PR in GitHub. Reads the PR comment. Follows the workbench deep link to review full evidence. Approves and merges the PR. The standard Terraform apply pipeline runs on merge.

**Agent:** None. Agent's work is complete. All outputs are in the PR and audit record.

**Data:**
- PR comment provides: failure class, evidence citations, validation status, workbench deep link
- Workbench deep link resolves: `https://app.terraform.io/workbench/sess-001` → session `sess-001` pre-loaded with full evidence panel, speculative run result, diff, and context sources
- GitHub standard PR review and merge flow; no new system calls from the agent

**Failure modes:**
- Workbench deep link returns session-not-found: session `sess-001` must be retained for 30 days minimum [A]. Surface "This session has expired. The PR description contains the full evidence summary."
- Approver merges without reading evidence: acceptable. The audit record captures what evidence was available and that the approver made the decision.

---

Admin sees in the audit view:
- Who triggered the agent and when
- What mode was active (L2 / Prepare PR)
- What the agent read (tools called, artifacts accessed)
- What the agent proposed (diff, unchanged vs. user-modified)
- What validation occurred (speculative run result, policy result)
- Who approved PR creation
- Whether PR was merged and by whom

**Validation:** Audit record must be immutable and queryable by run ID, workspace ID, session ID, and user identity. It must be retained beyond the session lifetime.

## Open Questions

- Is the `session_id` generated by the UI at button click, or by AGX at trigger receipt? Generation order affects the deep link timing in Stage 7. -- Engineering
- What is the exact API signature for the manual webhook trigger? The payload above is assumed from PDR-001; needs confirmation from AGX team. -- AGX team / Marc Cosentino
- Can the speculative run in Stage 6 be triggered by the agent directly via MCP, or does it require a separate HCP Terraform API call? If MCP, which tool? -- Engineering
- How is the PR description character limit handled if the evidence summary is long? GitHub's 65,536 character limit should not be a constraint for normal runs, but needs confirmation for large plan logs. -- Engineering
- Is the audit record written to the existing HCP Terraform run/workspace event log, or to a separate agent session store? Strategy doc 002 assumes the existing log; this needs engineering confirmation before PDR-003. -- Engineering
- When Alex edits the proposed diff (Stage 5), does the edited diff need to be recorded separately from the agent-proposed diff in the audit record? The scenario above assumes yes (`user_modified_diff: true`). -- Compliance / PM
